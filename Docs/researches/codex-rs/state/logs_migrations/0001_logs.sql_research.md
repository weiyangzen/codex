# 0001_logs.sql 研究文档

## 场景与职责

`0001_logs.sql` 是 Codex 项目日志数据库（logs database）的初始迁移脚本，负责创建用于存储应用日志的核心表结构。它是 `codex-rs/state` crate 中日志子系统的基石，为整个 Codex CLI/TUI 应用提供结构化的日志持久化能力。

该迁移脚本属于独立的日志迁移系统（`logs_migrations/` 目录），与主状态数据库迁移（`migrations/` 目录）分离，体现了架构上将日志存储与业务状态存储解耦的设计决策。

## 功能点目的

### 1. 核心表结构创建
创建 `logs` 表，用于存储应用运行期间产生的所有日志事件：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PRIMARY KEY AUTOINCREMENT | 自增主键，唯一标识每条日志 |
| `ts` | INTEGER NOT NULL | 日志时间戳（Unix 秒） |
| `ts_nanos` | INTEGER NOT NULL | 纳秒部分，提供微秒级精度 |
| `level` | TEXT NOT NULL | 日志级别（TRACE/DEBUG/INFO/WARN/ERROR） |
| `target` | TEXT NOT NULL | 日志目标模块路径 |
| `message` | TEXT | 日志消息内容（可为空） |
| `module_path` | TEXT | 源代码模块路径 |
| `file` | TEXT | 源代码文件名 |
| `line` | INTEGER | 源代码行号 |
| `thread_id` | TEXT | 关联的线程/对话 ID |
| `process_uuid` | TEXT | 进程唯一标识符 |
| `estimated_bytes` | INTEGER NOT NULL DEFAULT 0 | 预估字节数，用于分区容量控制 |

### 2. 索引优化
创建 5 个索引以支持高效的日志查询：

- **`idx_logs_ts`**: 按时间戳降序索引，支持时间范围查询和最新日志获取
- **`idx_logs_thread_id`**: 按线程 ID 索引，支持按对话/线程过滤
- **`idx_logs_process_uuid`**: 按进程 UUID 索引，支持按进程过滤
- **`idx_logs_thread_id_ts`**: 复合索引（thread_id + ts），优化特定线程的时间范围查询
- **`idx_logs_process_uuid_threadless_ts`**: 部分索引，仅包含 `thread_id IS NULL` 的记录，优化无线程日志的查询

## 具体技术实现

### 数据库设计决策

1. **时间戳双字段设计**: 使用 `ts`（秒）+ `ts_nanos`（纳秒）而非单一的高精度时间戳，便于与 Rust 的 `SystemTime` 和 `Duration` 类型直接映射，同时保持 SQLite 的整数比较效率。

2. **分区容量控制字段**: `estimated_bytes` 字段（后续迁移中引入）配合索引设计，支持按线程/进程维度的日志容量限制（10 MiB 分区上限）。

3. **可选字段设计**: `message`, `module_path`, `file`, `line`, `thread_id`, `process_uuid` 均为可空，允许在不同日志场景下的灵活使用。

### 关键代码路径

迁移脚本的执行路径：

```
codex-rs/state/src/runtime.rs:StateRuntime::init()
  └── open_sqlite(&logs_path, &LOGS_MIGRATOR)
        └── migrator.run(&pool)  // sqlx 迁移执行
              └── 执行 logs_migrations/ 目录下的 SQL 文件
```

迁移器定义在：
```rust
// codex-rs/state/src/migrations.rs
pub(crate) static LOGS_MIGRATOR: Migrator = sqlx::migrate!("./logs_migrations");
```

### 数据模型映射

Rust 中的 `LogEntry` 结构体（`codex-rs/state/src/model/log.rs`）与该表结构对应：

```rust
pub struct LogEntry {
    pub ts: i64,
    pub ts_nanos: i64,
    pub level: String,
    pub target: String,
    pub message: Option<String>,
    pub feedback_log_body: Option<String>,  // 在 0002 迁移中替代 message
    pub thread_id: Option<String>,
    pub process_uuid: Option<String>,
    pub module_path: Option<String>,
    pub file: Option<String>,
    pub line: Option<i64>,
}
```

插入操作位于 `codex-rs/state/src/runtime/logs.rs:insert_logs()`：
```rust

```

## 风险、边界与改进建议

### 已知风险

1. **schema 演进限制**: 作为初始迁移，后续 schema 变更需要通过新的迁移文件（如 0002_logs_feedback_log_body.sql）处理，需确保向后兼容。

2. **索引膨胀**: 5 个索引在大量日志写入场景下会增加写入开销，但读取性能优先的设计理念符合日志查询场景。

3. **部分索引限制**: `idx_logs_process_uuid_threadless_ts` 是部分索引（WHERE thread_id IS NULL），SQLite 版本需支持部分索引（3.8.0+）。

### 边界条件

- **时间戳范围**: `ts` 使用 INTEGER 存储 Unix 秒，可表示范围约为 ±68 年（以 1970 为基准）。
- **纳秒精度**: `ts_nanos` 存储纳秒部分（0-999999999），与 `ts` 组合提供完整精度。
- **自动递增**: `id` 使用 AUTOINCREMENT，保证单调递增但可能产生间隙（删除后不会复用）。

### 改进建议

1. **考虑添加 `created_at` 字段**: 虽然已有 `ts` 和 `ts_nanos`，但显式的 `created_at` 字段可简化查询。

2. **索引优化**: `idx_logs_process_uuid` 索引在 0002 迁移后实际上未被使用（被部分索引替代），可考虑移除。

3. **分区策略文档化**: `estimated_bytes` 字段的用途和分区容量控制逻辑应在 schema 注释中说明。

4. **考虑日志级别索引**: 如果频繁按级别过滤大量日志，可考虑添加 `idx_logs_level`。
