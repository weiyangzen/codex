# 0002_logs.sql 研究文档

## 场景与职责

本迁移文件创建 `logs` 表，用于存储 Codex 应用程序的日志数据。这是日志持久化存储的基础表，支持后续的日志查询、调试、反馈收集等功能。

## 功能点目的

### 1. logs 表结构
创建包含以下字段的日志表：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PRIMARY KEY AUTOINCREMENT | 自增主键 |
| `ts` | INTEGER NOT NULL | 时间戳（Unix秒） |
| `ts_nanos` | INTEGER NOT NULL | 时间戳纳秒部分 |
| `level` | TEXT NOT NULL | 日志级别（TRACE/DEBUG/INFO/WARN/ERROR） |
| `target` | TEXT NOT NULL | 日志目标模块 |
| `message` | TEXT | 日志消息内容 |
| `module_path` | TEXT | 模块路径 |
| `file` | TEXT | 源文件名 |
| `line` | INTEGER | 行号 |

### 2. 索引设计
- `idx_logs_ts`: 按时间戳倒序查询，支持日志流式读取

## 具体技术实现

### 关键流程
1. **日志写入**: 通过 `tracing` 集成，异步批量写入
2. **时间精度**: 使用秒+纳秒分离存储，支持微秒级精度
3. **自增ID**: 使用 SQLite 的 AUTOINCREMENT 确保单调递增

### 日志查询
- `codex-rs/state/src/runtime/logs.rs`:

- 默认保留 10 天（`LOG_RETENTION_DAYS = 10`）
- 定期清理通过 `run_retention_cleanup` 执行

## 风险、边界与改进建议

### 风险
1. **存储膨胀**: 高频日志可能导致数据库快速增长
2. **写入性能**: 大量并发日志写入可能成为瓶颈

### 边界情况
1. **时间回拨**: 系统时间回拨可能导致 `ts` 重复，依赖 `id` 保证顺序
2. **纳秒溢出**: `ts_nanos` 应始终小于 1_000_000_000

### 改进建议
1. 已实施：后续迁移添加了 `estimated_bytes` 和分区裁剪机制
2. 考虑按日期分区表（如果日志量极大）
3. 可为 `level` 字段添加索引（如果频繁按级别筛选）
