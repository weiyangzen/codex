## Component Documentation

### ColumnWidthMode Enum
```rust
#[derive(Clone, Copy, Debug, Default, PartialEq, Eq)]
pub(crate) enum ColumnWidthMode {
    /// Derive column placement from only the visible viewport rows.
    #[default]
    AutoVisible,
    /// Derive column placement from all rows so scrolling does not shift columns.
    AutoAllRows,
    /// Use a fixed two-column split: 30% left (name), 70% right (description).
    Fixed,
}
```

### Key Functions

- `render_rows_stable_col_widths()` - Renders rows with stable column widths

### Rendering Flow
1. `desired_height()` calls `measure_rows_height_stable_col_widths()` for `AutoAllRows`
2. Column widths are computed by measuring ALL rows, not just visible viewport
3. When scrolling, the name column width remains constant because it accounts for the longest item
4. This prevents column "jitter" when scrolling through items of varying lengths

## Architectural Notes
The `ColumnWidthMode` enum provides three strategies for handling column widths in selection lists:

1. **AutoVisible** (default): Measures only visible rows for efficiency. Column widths may shift when scrolling if off-screen items have different lengths.

2. **AutoAllRows**: Measures all rows to ensure stable column widths during scrolling. Trade-off: slightly more computation upfront, but better UX for lists with varying item lengths.

3. **Fixed**: Uses a fixed 30/70 split between name and description columns. Predictable but may waste space or cause truncation.

This snapshot demonstrates `AutoAllRows` behavior where the column width is determined by "Item 9 with an intentionally much longer name" even when it's not visible, ensuring the layout doesn't shift when the user scrolls to it.

The test validates that:
- Column widths remain consistent before and after scrolling
- The longest item (item 9) affects layout even when off-screen
- Selection indicator (›) moves correctly with scrolling
- Number prefixes update to reflect visible items
