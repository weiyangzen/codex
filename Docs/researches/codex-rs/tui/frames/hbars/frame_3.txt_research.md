# frame_3.txt 研究文档

## 场景与职责

`frame_3.txt` 是 Codex TUI 欢迎界面 `hbars` 动画变体的第 3 帧。该帧继续展示水平条形图的动态演变，在 36 帧循环序列中位于约 160ms 时间点。

本帧在动画序列中起到承上启下的作用，条形图案从 frame_2.txt 的状态继续演化，保持视觉流动的连贯性。

## 功能点目的

1. **动画平滑过渡**：作为第 3 帧，延续前两帧建立的视觉节奏
2. **视觉多样性**：通过条形高度和位置的变化，避免静态图案的单调感
3. **用户等待反馈**：在 Codex 初始化或登录过程中提供视觉反馈，减少用户感知的等待时间

## 具体技术实现

### 数据结构

- **文件大小**：1218 字节
- **字符分布**：
  - 高密度区域：中心偏上位置，使用 `█`、`▇`、`▆` 等完整/近完整块字符
  - 中密度区域：过渡带，使用 `▅`、`▄`、`▃` 等中等高度块字符
  - 低密度区域：边缘，使用 `▂`、`▁` 等低位块字符或空格

### 关键流程

```rust
// 动画渲染流程（简化）
1. WelcomeWidget::render_ref() 被调用
2. 检查 animations_enabled && 终端尺寸 >= 最小要求
3. AsciiAnimation::current_frame() 计算当前帧索引
4. 根据时间偏移选择 frame_3.txt（当 elapsed_ms 在 160-240ms 之间）
5. 将帧内容按行分割并转换为 ratatui::text::Line
6. 渲染到终端缓冲区
```

### 帧切换时机

| 时间点 | 活动帧 | 帧索引 |
|--------|--------|--------|
| 0ms | frame_1.txt | 0 |
| 80ms | frame_2.txt | 1 |
| 160ms | frame_3.txt | 2 |
| 240ms | frame_4.txt | 3 |

## 依赖与外部交互

### 上游调用链

```
main.rs → Tui::run() → event loop
    ↓
WelcomeWidget::render_ref()
    ↓
AsciiAnimation::current_frame()
    ↓
FRAMES_HBARS[2] // frame_3.txt 内容
```

### 外部配置

- **动画开关**：通过 `animations_enabled` 参数控制
- **最小尺寸**：`MIN_ANIMATION_HEIGHT = 37`, `MIN_ANIMATION_WIDTH = 60`
- **帧间隔**：`FRAME_TICK_DEFAULT = 80ms`

### 平行实现

`tui_app_server` crate 中 `frames.rs` 包含完全相同的帧数据，确保服务器端渲染一致性。

## 风险、边界与改进建议

### 潜在问题

1. **字符对齐**：某些终端字体中 Unicode 块字符宽度可能与空格不一致，导致图案错位
2. **性能考虑**：虽然单次渲染开销小，但 80ms 的频繁重绘在远程终端（SSH）上可能产生可见的带宽消耗
3. **可访问性**：动画可能对某些用户造成分心或不适（前庭功能障碍）

### 边界条件

- **终端缩放**：用户调整终端大小时，动画可能突然显示/隐藏
- **焦点丢失**：终端失去焦点时动画继续运行（当前实现），可能浪费资源
- **多显示器**：在不同 DPI 的显示器间移动终端窗口无特殊处理

### 改进建议

1. **动画暂停**：当终端失去焦点时暂停动画，节省资源
2. **减少运动**：提供 `prefers-reduced-motion` 选项，完全禁用动画
3. **帧率自适应**：根据终端连接类型（本地/远程）动态调整帧率
4. **缓存优化**：预计算所有帧的 `Vec<Line>` 避免重复解析

### 代码质量

- **类型安全**：使用 `&'static str` 确保帧数据生命周期安全
- **越界保护**：`% frames.len()` 确保索引安全
- **测试覆盖**：现有测试验证动画渲染但不验证具体帧内容
