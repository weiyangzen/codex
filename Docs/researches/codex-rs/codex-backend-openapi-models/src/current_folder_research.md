## 场景与职责

`src` 目录是 `codex-backend-openapi-models` 的唯一源码目录，职责是承载后端 OpenAPI 数据模型（DTO），并以最小导出面供 workspace 其他 crate 复用。

该目录的职责边界非常明确：

1. 模型承载，不承担网络调用
- `src/lib.rs` 只暴露 `pub mod models;`，并明确“本模块不包含手写业务类型”（`codex-rs/codex-backend-openapi-models/src/lib.rs:3-6`）。

2. OpenAPI 生成产物容器
- `src/models/*.rs` 全部带 OpenAPI Generator 头注释（如 `Generated by: https://openapi-generator.tech`），说明来源主要是自动生成结果（示例：`.../src/models/rate_limit_status_payload.rs:1-9`）。

3. 受控导出面
- `src/models/mod.rs` 不是“全量自动导出”，而是“当前 workspace 实际用到类型的 curated 清单”（`.../src/models/mod.rs:1-4`）。

4. 与 workspace 的职责分层
- 本目录负责“结构定义”；真正的协议归一化、字段语义修正、业务映射在调用方 `backend-client`/`cloud-tasks-client` 完成。

## 功能点目的

### 1) 配置文件响应模型（Config）

目的：表达云端 requirements 配置文件下载接口的响应结构。

- 类型：`ConfigFileResponse`（`.../src/models/config_file_response.rs:14-24`）
- 关键字段：`contents`、`sha256`、`updated_at`、`updated_by_user_id`
- 使用目标：`backend-client` 的 `get_config_requirements_file()` 反序列化目标（`codex-rs/backend-client/src/client.rs:347-357`），上层由 `cloud-requirements` 消费 `contents`（`codex-rs/cloud-requirements/src/lib.rs:220-244`）。

### 2) 任务域模型（Cloud Tasks）

目的：承载任务列表、任务详情、外部 PR 相关结构。

- `TaskListItem`（`.../src/models/task_list_item.rs:16-41`）：列表项基础模型
- `PaginatedListTaskListItem`（`.../src/models/paginated_list_task_list_item_.rs:16-21`）：分页列表包装
- `TaskResponse`（`.../src/models/task_response.rs:16-41`）：任务详情中的任务主结构
- `CodeTaskDetailsResponse`（`.../src/models/code_task_details_response.rs:16-31`）：详情响应包装（生成版）
- `ExternalPullRequestResponse` / `GitPullRequest`（`.../src/models/external_pull_request_response.rs:16-25`, `.../src/models/git_pull_request.rs:15-48`）

补充：`backend-client` 明确认为生成版 task-details 模型可用性较差，改用手写结构（`codex-rs/backend-client/src/types.rs:16-20`）。因此该目录中的 `CodeTaskDetailsResponse` 更多是“契约镜像”，不是业务解析主实现。

### 3) 配额域模型（Rate Limits）

目的：承载 `/usage` 返回的计划、主配额、附加配额、credit 信息。

- `RateLimitStatusPayload`（`.../src/models/rate_limit_status_payload.rs:16-40`）
- `RateLimitStatusDetails`（`.../src/models/rate_limit_status_details.rs:16-35`）
- `RateLimitWindowSnapshot`（`.../src/models/rate_limit_window_snapshot.rs:14-23`）
- `AdditionalRateLimitDetails`（`.../src/models/additional_rate_limit_details.rs:16-28`）
- `CreditStatusDetails`（`.../src/models/credit_status_details.rs:14-40`）
- `PlanType` 枚举（`.../src/models/rate_limit_status_payload.rs:56-84`）

目的不是直接供 UI 用，而是先由 `backend-client` 映射成 `codex_protocol::RateLimitSnapshot`（`codex-rs/backend-client/src/client.rs:395-494`）。

### 4) 导出治理功能

`models/mod.rs` 把导出分为 `Config / Cloud Tasks / Rate Limits` 三组（`.../src/models/mod.rs:6-44`），作用是：

- 限制依赖方看到的类型集合
- 降低模型再生时对 workspace 的突发影响
- 作为“实际使用面”的人工闸门

## 具体技术实现（关键流程/数据结构/协议/命令）

### A. 模块组织与构建落地

1. 模块结构

3. Bazel 接入

### B. 关键数据结构设计

1. 三态字段：`Option<Option<T>>` + `serde_with::rust::double_option`
- 典型字段：
  - `RateLimitStatusPayload.rate_limit`（`.../rate_limit_status_payload.rs:19-25`）
  - `RateLimitStatusPayload.credits`（`.../rate_limit_status_payload.rs:26-33`）
  - `RateLimitStatusDetails.primary_window`（`.../rate_limit_status_details.rs:21-28`）
  - `CreditStatusDetails.balance`（`.../credit_status_details.rs:19-25`）
- 语义价值：区分“缺省字段 / 显式 null / 有值”。

2. 弱类型字段保留后端灵活性
- `TaskListItem.task_status_display` 是 `HashMap<String, Value>`（`.../task_list_item.rs:30-35`）
- `GitPullRequest.comments/diff/user` 为 `serde_json::Value`（`.../git_pull_request.rs:42-47`）
- 这让模型对后端变更更容忍，但把语义解释压力转给上层。

3. 嵌套关系
- `TaskResponse` -> `Vec<ExternalPullRequestResponse>`（`.../task_response.rs:39-40`）
- `ExternalPullRequestResponse` -> `GitPullRequest`（`.../external_pull_request_response.rs:21-23`）
- `RateLimitStatusPayload` -> `RateLimitStatusDetails`/`CreditStatusDetails`/`AdditionalRateLimitDetails`（`.../rate_limit_status_payload.rs:25,32,39`）

### C. 关键流程（调用链视角）

虽然该目录只定义模型，但处在请求链路核心：

1. Usage 配额流程
- `backend-client` 请求 `/api/codex/usage` 或 `/wham/usage`，先解码为 `RateLimitStatusPayload`（`codex-rs/backend-client/src/client.rs:257-265`）
- 然后映射为 `RateLimitSnapshot`（`.../client.rs:395-494`）
- 上层消费：
  - app-server：`fetch_account_rate_limits()`（`codex-rs/app-server/src/codex_message_processor.rs:1466-1532`）
  - tui：`fetch_rate_limits()`（`codex-rs/tui/src/chatwidget.rs:9489-9502`）

2. Tasks 列表流程
- `backend-client::list_tasks()` 把响应解码为 `PaginatedListTaskListItem`（`codex-rs/backend-client/src/client.rs:268-302`）
- `cloud-tasks-client` 再映射 `TaskListItem -> TaskSummary`（`codex-rs/cloud-tasks-client/src/http.rs:145-177,714-730`）

3. Task details 流程
- 目录中存在生成版 `CodeTaskDetailsResponse`（`.../code_task_details_response.rs:16-31`）
- 但实际主路径使用 `backend-client` 手写 `CodeTaskDetailsResponse`（`codex-rs/backend-client/src/types.rs:16-27`）并提供 `CodeTaskDetailsResponseExt` 用于 diff / messages / prompt 抽取（`.../types.rs:260-300`）

4. 云端 requirements 流程
- `backend-client::get_config_requirements_file()` 反序列化 `ConfigFileResponse`（`codex-rs/backend-client/src/client.rs:347-357`）
- `cloud-requirements` 读取 `response.contents` 并处理 unauthorized/retry（`codex-rs/cloud-requirements/src/lib.rs:220-244`）

### D. 协议与命令接口（由调用方触发）

本目录模型主要对应这些 HTTP 接口：

- `GET {base}/api/codex/usage` 或 `GET {base}/wham/usage`
- `GET {base}/api/codex/tasks/list` 或 `GET {base}/wham/tasks/list`
- `GET {base}/api/codex/tasks/{task_id}` 或 `GET {base}/wham/tasks/{task_id}`
- `GET {base}/api/codex/config/requirements` 或 `GET {base}/wham/config/requirements`

路径风格由 `backend-client::PathStyle` 决定（`codex-rs/backend-client/src/client.rs:257-277,313-353`）。

### E. 脚本与再生流程现状

- `src/lib.rs` 注释提到“regen script 会写 `src/models/*.rs` 和 `src/models/mod.rs`”（`.../src/lib.rs:3-4`）。
- 但当前仓库内没有发现该 crate 专属的明确再生入口脚本（以 `rg` 检索 `openapi/generator/regen` 结果看，更多是注释与间接说明）。
- 这意味着当前是“生成产物 + 人工 curated 导出”混合维护形态。

## 依赖与外部交互

### 1) 目录内直接依赖

- `serde`：`Serialize`/`Deserialize` derive
- `serde_json`：`Value` 动态字段
- `serde_with`：`double_option` 三态字段语义

### 2) 与外部系统交互的边界

- 本目录本身不发起 HTTP、不访问文件系统、不读环境变量。
- 与后端 API 的真实交互在 `backend-client` 执行，请求返回 JSON 后再交由本目录模型反序列化。

### 3) 上下游协议耦合

- 上游（后端）通过 OpenAPI 结构驱动字段名和可空性。
- 下游（`backend-client`、`cloud-tasks-client`、`app-server`、`tui`）依赖这些字段进行映射和展示。
- `PlanType` 存在模型层枚举值到协议层 `AccountPlanType` 的二次映射（`codex-rs/backend-client/src/client.rs:470-485`）。

## 风险、边界与改进建议

### 风险 1：生成模型与手写模型并存造成语义分叉

现象：`CodeTaskDetailsResponse` 在本目录是生成版 map 结构（`.../code_task_details_response.rs:16-31`），但实际业务路径使用 `backend-client` 手写结构（`codex-rs/backend-client/src/types.rs:16-27`）。

影响：
- 同名结构在不同 crate 含义不同，容易误用。

建议：
- 在 `models/mod.rs` 或对应模型文件加注释，标注该类型“仅契约镜像，不建议业务直接消费”。
- 中期可考虑从导出列表中移除该类型，减少歧义。

### 风险 2：`Option<Option<T>>` 扩散导致调用复杂度高

现象：三态字段在 rate-limit 相关模型普遍存在（如 `.../rate_limit_status_payload.rs:19-39`）。

影响：
- 调用方容易遗漏 `flatten()`，出现“null/缺省”判定错误。

建议：
- 在 `backend-client` 增加统一归一化层，把三态折叠为业务可直接消费的 canonical 结构。
- 在该目录增加注释文档明确三态语义。

### 风险 3：弱类型 `serde_json::Value` 字段带来运行时风险

现象：`task_status_display`、`comments/diff/user` 等字段为动态 JSON（`.../task_list_item.rs:30-35`, `.../git_pull_request.rs:42-47`）。

影响：
- 编译期无约束，字段漂移只能在运行期暴露。

建议：
- 对稳定子结构逐步引入强类型（可从 `task_status_display.latest_turn_status_display` 开始）。
- 保留动态字段作为 fallback，分阶段迁移。

### 风险 4：时间字段使用 `i32`

现象：`RateLimitWindowSnapshot.reset_at` 为 `i32`（`.../rate_limit_window_snapshot.rs:21-22`）。

影响：
- 潜在 2038 边界问题，向协议层转换时虽然转为 `i64`，但源头仍受限。

建议：
- 与后端契约确认后升级为 `i64`，并同步调用方测试。

### 风险 5：再生流程文档不显式

现象：`lib.rs` 提及 regen script（`.../src/lib.rs:3-4`），但缺少可直接执行的、目录级可见的再生说明。

影响：
- 维护者在模型变更时容易漏掉导出清单/再生步骤。

建议：
- 在 `codex-backend-openapi-models` 目录补一个简短 README，写清：输入 spec、生成命令、`models/mod.rs` 维护规则、验证命令。

### 边界说明

- 本目录是“数据契约层”，不是“业务语义层”。
- 业务转换（diff 抽取、配额快照映射、错误处理）应继续集中在 `backend-client` 及其上层，避免把逻辑回流到模型目录。
