## 场景与职责

`src/models` 是 Codex Rust workspace 中“后端 OpenAPI 契约 DTO 层”的核心目录，职责是把后端 JSON 结构稳定地映射成 Rust 可反序列化类型，供上层客户端与业务层消费。

该目录的职责边界很清晰：

1. 只承载数据结构，不执行业务逻辑
- 目录内文件全部是 `struct/enum + serde 注解 + new 构造器`；无 HTTP、无文件 I/O、无环境变量读取。
- 入口 `lib.rs` 仅暴露 `pub mod models;`，并说明这里是生成模型再导出层：`codex-rs/codex-backend-openapi-models/src/lib.rs:3-6`。

2. 承接 OpenAPI 生成结果
- 每个模型文件头注释均包含 `Generated by: https://openapi-generator.tech`：如 `.../src/models/rate_limit_status_payload.rs:1-9`、`.../src/models/task_response.rs:1-9`。

3. 提供受控导出面
- `mod.rs` 明确“当前只导出 workspace 实际使用的最小集合”，属于人工 curated 导出，而非全自动全量导出：`codex-rs/codex-backend-openapi-models/src/models/mod.rs:1-4`。

4. 与调用方分层
- 本目录负责“契约结构承载”。
- 业务语义转换在 `backend-client` / `cloud-tasks-client` 完成（例如 rate limit 归一化、task details 富解析）：`codex-rs/backend-client/src/client.rs:395-494`、`codex-rs/backend-client/src/types.rs:16-305`。

## 功能点目的

### 1) Config 模型：云端 requirements 配置文件响应

- 类型：`ConfigFileResponse`（`.../config_file_response.rs:14-24`）
- 目的：承载 `GET /config/requirements` 的 `contents/sha256/updated_at/updated_by_user_id`。
- 使用路径：
  - `backend-client` 直接反序列化：`codex-rs/backend-client/src/client.rs:347-357`
  - `cloud-requirements` 读取 `contents` 并进入解析/重试流程：`codex-rs/cloud-requirements/src/lib.rs:220-244`

### 2) Cloud Tasks 模型：任务列表、详情、PR 元信息

- 列表：`TaskListItem`（`.../task_list_item.rs:15-41`）
- 分页：`PaginatedListTaskListItem`（`.../paginated_list_task_list_item_.rs:15-21`）
- 详情主体：`TaskResponse`（`.../task_response.rs:15-41`）
- 详情包装：`CodeTaskDetailsResponse`（`.../code_task_details_response.rs:15-31`）
- PR 相关：`ExternalPullRequestResponse`、`GitPullRequest`（`.../external_pull_request_response.rs:15-25`、`.../git_pull_request.rs:14-48`）

目标是提供“原始契约结构”。
补充事实：任务详情实际主消费并不直接使用该目录的生成版 `CodeTaskDetailsResponse`，而是 `backend-client` 自定义手写版本（注明“generated models are pretty bad”）：`codex-rs/backend-client/src/types.rs:16-20`。

### 3) Rate Limits 模型：套餐、窗口、信用额度与附加配额

- 顶层：`RateLimitStatusPayload`（`.../rate_limit_status_payload.rs:15-40`）
- 明细：`RateLimitStatusDetails`（`.../rate_limit_status_details.rs:15-35`）
- 窗口：`RateLimitWindowSnapshot`（`.../rate_limit_window_snapshot.rs:13-23`）
- 附加配额：`AdditionalRateLimitDetails`（`.../additional_rate_limit_details.rs:15-28`）
- 额度：`CreditStatusDetails`（`.../credit_status_details.rs:13-40`）
- 套餐枚举：`PlanType`（`.../rate_limit_status_payload.rs:53-84`）

目标是承载 `/usage` 接口原始响应，随后由 `backend-client` 映射为 `codex_protocol::RateLimitSnapshot`：`codex-rs/backend-client/src/client.rs:257-266,395-494`。

### 4) 导出治理功能

`mod.rs` 将模型分组为 Config / Cloud Tasks / Rate Limits 并手工 re-export（`.../mod.rs:6-44`），目的：

- 限制 API 面，避免生成器全量暴露；
- 把“实际被 workspace 使用”的集合显式化；
- 降低后端契约突变时的横向影响。

## 具体技术实现（关键流程/数据结构/协议/命令）

### A. 数据结构实现机制

1. 统一序列化注解策略
- 目录内模型统一使用 `#[derive(Serialize, Deserialize)]`。
- 字段命名通过 `#[serde(rename = "...")]` 对齐后端 wire key，例如 `plan_type`、`external_pull_requests`：`.../rate_limit_status_payload.rs:17`、`.../task_response.rs:39-40`。

2. 三态可空语义（关键）
- 多个字段采用 `Option<Option<T>> + serde_with::rust::double_option`，例如：
  - `RateLimitStatusPayload.rate_limit`：`.../rate_limit_status_payload.rs:19-25`
  - `RateLimitStatusPayload.credits`：`.../rate_limit_status_payload.rs:26-33`
  - `RateLimitStatusDetails.primary_window`：`.../rate_limit_status_details.rs:21-27`
  - `CreditStatusDetails.balance`：`.../credit_status_details.rs:19-25`
- 该形态可区分：字段缺省 / 显式 `null` / 有值。
- 调用方消费时需显式 `flatten()`，见 `backend-client`：`codex-rs/backend-client/src/client.rs:403-404,412,448,466`。

3. 弱类型字段保留协议弹性
- 任务状态与 PR 部分字段采用 `serde_json::Value` / `HashMap<String, Value>`：
  - `TaskListItem.task_status_display`：`.../task_list_item.rs:30-35`
  - `GitPullRequest.comments/diff/user`：`.../git_pull_request.rs:42-47`
- 代价是编译期类型约束不足，语义解释转移到业务层。

### B. 关键流程（调用链）

1. Rate limits 读取与归一化
- `backend-client` 请求：`/api/codex/usage` 或 `/wham/usage`：`codex-rs/backend-client/src/client.rs:257-261`
- 先解码为 `RateLimitStatusPayload`：`.../client.rs:264`
- 再映射为协议层 `RateLimitSnapshot`：`.../client.rs:395-494`
- 计划类型再映射到 `codex_protocol::account::PlanType`：`.../client.rs:470-485`，协议定义见 `codex-rs/protocol/src/account.rs:6-20`。
- 下游继续透传到 app-server v2：`codex-rs/app-server-protocol/src/protocol/v2.rs:5723-5745`。

2. 任务列表流程
- `backend-client::list_tasks()` 反序列化 `PaginatedListTaskListItem`：`codex-rs/backend-client/src/client.rs:268-302`
- `cloud-tasks-client` 将 `TaskListItem` 映射为领域 `TaskSummary`：`codex-rs/cloud-tasks-client/src/http.rs:145-177,714-730`。
- `pull_requests` 非空即标记 `is_review`：`.../http.rs:724-727`，来源字段是模型中的 `pull_requests`（`.../task_list_item.rs:39-40`）。

3. 任务详情流程（生成模型与手写模型分流）
- OpenAPI 目录提供生成版 `CodeTaskDetailsResponse`（`.../code_task_details_response.rs:15-31`）。
- 实际请求解析走 `backend-client` 手写模型与扩展 trait：
  - 类型定义：`codex-rs/backend-client/src/types.rs:16-117`
  - diff/messages/prompt/error 提取：`.../types.rs:260-305`
  - `cloud-tasks-client` 消费这些扩展能力：`codex-rs/cloud-tasks-client/src/http.rs:249-314,437-443`

4. 云端 requirements 配置流程
- `backend-client::get_config_requirements_file()` 解码 `ConfigFileResponse`：`codex-rs/backend-client/src/client.rs:347-357`
- `cloud-requirements` 在鉴权、重试、缓存流程中消费 `contents`：`codex-rs/cloud-requirements/src/lib.rs:205-244,326-357,369-510`。

### C. 协议与外部接口（由调用方触发）

## 风险、边界与改进建议

### 风险 1：生成模型与手写模型并存，语义分叉风险高

- 现象：同名“任务详情”在 openapi-models 与 backend-client 语义不同（生成版 map vs 手写强语义提取）。
- 证据：`.../code_task_details_response.rs:16-31` vs `codex-rs/backend-client/src/types.rs:16-305`。
- 影响：新调用方可能误选生成版，导致能力不足（无 `unified_diff`/`assistant_text_messages` 等）。
- 建议：
  1. 在 `models/mod.rs` 为 `CodeTaskDetailsResponse` 增加注释，标注“契约镜像，不建议直接业务消费”；
  2. 在 `backend-client` 文档中明确“任务详情只能走 hand-rolled model + ext trait”。

### 风险 2：`Option<Option<T>>` 三态传播使调用点复杂、易误判

- 现象：rate-limit 相关字段普遍三态；调用方大量 `flatten()`。
- 影响：遗漏 flatten 或误解“缺省 vs null”将导致逻辑偏差。
- 建议：
  1. 在 `backend-client` 增加单点归一化函数并集中测试三态边界；
  2. 在本目录补简短注释文档，明确三态语义表。

### 风险 3：弱类型字段（`serde_json::Value`）降低编译期保障

- 现象：`task_status_display`、`GitPullRequest.diff/comments/user` 为动态 JSON。
- 影响：后端字段漂移只能在运行期暴露，且错误定位晚。
- 建议：
  1. 针对稳定子结构（如 `latest_turn_status_display`、`diff_stats`）引入渐进式强类型；
  2. 保留原 `Value` fallback 字段以保持向后兼容。

### 风险 4：时间与数值类型边界不统一

- 现象：
  - `TaskListItem.updated_at/created_at` 使用 `Option<f64>`（秒级浮点）；
  - `RateLimitWindowSnapshot.reset_at` 使用 `i32`。
- 影响：精度与范围策略分散，映射链存在隐式截断/转换。
- 建议：
  1. 在业务归一化层统一为 `i64 Unix seconds`；
  2. 为跨层转换增加边界测试（负值、超大值、分数秒）。

### 风险 5：本目录无独立测试，回归依赖下游

- 现象：`src/models` 无 `tests/`；当前验证主要依赖 `backend-client` 测试。
- 影响：模型字段变更后，问题可能在更远的业务测试中才暴露。
- 建议：
  1. 至少为关键模型增加“反序列化 smoke tests”（最小 fixture）；
  2. 对 `PlanType`、`double_option` 字段加契约测试，降低后端字段变更风险。

### 风险 6：regen 流程可见性不足

- 现象：代码注释提及“regen script”，但仓库未见该 crate 清晰入口。
- 影响：后续维护者难以标准化更新模型，容易手工改动漂移。
- 建议：
  1. 在 `codex-rs/codex-backend-openapi-models/` 增加 README，明确 spec 来源、regen 命令、`mod.rs` 维护规则、验证命令；
  2. 若流程已存在于其他仓库或私有脚本，至少在本仓库补跳转说明。
