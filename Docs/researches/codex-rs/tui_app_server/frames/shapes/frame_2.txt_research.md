# shapes/frame_2.txt 研究文档

## 场景与职责

`shapes/frame_2.txt` 是 Codex TUI 应用程序服务器的 ASCII 艺术动画帧文件，属于 `shapes`（形状）动画变体的第 2 帧。作为 36 帧循环动画序列的一部分，它与相邻帧（frame_1.txt 和 frame_3.txt）共同构成流畅的视觉过渡效果。

**使用场景：**
- TUI 欢迎界面的连续动画播放
- 作为 shapes 变体 36 帧序列中的第 2 帧，在 80ms 间隔后接替 frame_1 显示
- 用户通过 `Ctrl+.` 切换到 shapes 变体时可能从任意帧开始显示

## 功能点目的

1. **动画连续性**：与 frame_1 形成视觉过渡，创造动态效果
2. **视觉节奏**：通过形状位置的微小变化产生"流动"感
3. **品牌体验**：shapes 变体使用几何形状符号，传达现代、抽象的科技感

## 具体技术实现

### 帧内容特征
```
帧 2 特征分析：
- 使用符号：◆、△、●、□、▲、◇、○、■（与 frame_1 相同字符集）
- 图案演变：相比 frame_1，形状分布更加集中，中心区域密度增加
- 视觉变化：外围形状向中心收缩，形成"凝聚"的过渡效果
```

### 动画时序
```rust
// ascii_animation.rs
pub(crate) fn current_frame(&self) -> &'static str {
    let frames = self.frames();
    let elapsed_ms = self.start.elapsed().as_millis();
    let idx = ((elapsed_ms / tick_ms) % frames.len() as u128) as usize;
    frames[idx]  // 当 idx == 1 时返回 frame_2.txt 内容
}
```

### 帧调度
- 第 2 帧在第 80-159ms 时间窗口显示
- 由 `FrameRequester` 调度，确保 80ms 间隔的准确渲染

## 依赖与外部交互

### 上游依赖
- `frame_1.txt`：视觉上的前序帧，本帧继承其视觉状态并演变

### 下游依赖
- `frame_3.txt`：视觉上的后续帧，本帧的状态将过渡到下一帧

## 风险、边界与改进建议

### 动画一致性风险
- **风险**：如果帧之间差异过大，会产生"跳跃"感而非流畅动画
- **缓解**：当前 frame_2 与 frame_1 的形状分布保持连贯，过渡自然

### 性能考虑
- 作为静态字符串嵌入，访问时间为 O(1)
- 无运行时文件 I/O，渲染延迟仅取决于终端输出速度

### 改进建议
1. **帧插值**：考虑在运行时进行帧间插值，减少所需帧数
2. **自适应帧率**：根据终端性能动态调整帧间隔
3. **帧验证工具**：添加 CI 检查确保所有变体的帧数一致（均为 36 帧）
