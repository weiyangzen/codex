# FsCopyResponse.ts Research Document

## 场景与职责

`FsCopyResponse` 是 Codex App-Server 协议 v2 中文件系统 RPC 方法 `fs/copy` 的成功响应类型。它表示文件或目录复制操作已成功完成，是 `FsCopyParams` 请求的预期响应。

主要应用场景包括：
- **复制确认**: 向调用方确认复制操作已成功执行
- **流程控制**: 作为异步操作链中的成功信号，触发后续操作
- **错误边界**: 与错误响应区分，明确操作状态

## 功能点目的

该类型的设计目标是提供一种简洁的、类型安全的方式来表示复制操作的成功状态：

1. **空响应模式**: 使用 `Record<string, never>` 表示操作成功但无需返回额外数据
2. **类型安全**: TypeScript 类型系统确保响应结构的一致性
3. **协议一致性**: 遵循 App-Server v2 API 设计规范，所有方法都有明确定义的响应类型
4. **未来扩展性**: 空对象结构为未来添加可选字段预留空间

#### Record<string, never> 详解

`Record<string, never>` 是 TypeScript 中表示"空对象"的惯用类型：
- `Record<K, V>` 表示键类型为 `K`、值类型为 `V` 的对象
- `never` 表示不可能存在的值
- 因此 `Record<string, never>` 表示一个不能有任何属性的对象

这与其他空对象表示法的区别：

| 类型 | 含义 | 可接受值 |
|------|------|----------|
| `Record<string, never>` | 严格空对象 | `{}` |
| `{}` | 任何非 null/undefined 值 | `{}`, `[]`, 函数等 |
| `object` | 任何对象 | `{}`, `[]`, `{a: 1}` 等 |
| `void` | 无返回值 | 通常用于函数返回类型 |

## 风险、边界与改进建议

### 潜在风险

1. **信息不足**: 空响应不提供任何关于复制操作的信息（如复制的字节数、文件数）
2. **调试困难**: 出现问题时，无法从响应中判断复制是否部分完成
3. **幂等性确认**: 无法区分"复制成功"和"目标已存在且内容相同"

### 边界情况

| 场景 | 当前行为 | 潜在问题 |
|------|----------|----------|
| 复制成功 | 返回 `{}` | 无 |
| 部分成功 | 返回错误 | 无法知道哪些文件已复制 |
| 源和目标相同 | 取决于实现 | 可能静默成功或报错 |

### 改进建议

1. **添加基本统计信息**:
   ```typescript
   export type FsCopyResponse = {
     /** 复制的文件数量 */
     filesCopied?: number;
     /** 复制的字节数 */
     bytesCopied?: number;
     /** 操作耗时（毫秒） */
     durationMs?: number;
   };
   ```

2. **添加目标路径确认**:
   ```typescript
   export type FsCopyResponse = {
     /** 实际的目标路径（可能与请求不同，如自动重命名） */
     actualDestinationPath: AbsolutePathBuf;
   };
   ```

3. **支持详细模式**:
   ```typescript
   // 在 FsCopyParams 中添加
   export type FsCopyParams = {
     // ... 现有字段
     verbose?: boolean;  // 请求详细响应
   };

   // 详细响应
   export type FsCopyVerboseResponse = {
     copiedFiles: Array<{
       source: string;
       destination: string;
       size: number;
     }>;
     skippedFiles: string[];  // 因已存在而跳过的文件
     errors: Array<{
       path: string;
       error: string;
     }>;
   };
   ```

4. **添加校验和**:
   ```typescript
   export type FsCopyResponse = {
     /** 复制后目标文件的校验和 */
     checksum?: string;
     /** 校验和算法 */
     checksumAlgorithm?: "sha256" | "md5";
   };
   ```

### 设计考量

#### 为什么选择空响应？

1. **简单性**: 复制操作是"执行并忘记"模式，调用方通常只关心成功/失败
2. **性能**: 减少响应体大小，提高传输效率
3. **一致性**: 与其他简单文件操作（如 `fs/createDirectory`）保持一致
4. **YAGNI**: 当前没有明确的用例需要返回额外信息

#### 何时应该添加字段？

当出现以下需求时，应考虑扩展响应类型：
- 需要向用户展示复制统计信息
- 需要确认复制的完整性（校验和）
- 需要支持部分成功场景
- 需要审计和日志记录

### 最佳实践

- 客户端收到空响应后，应视为操作成功完成
- 如需确认复制结果，可随后调用 `fs/getMetadata` 检查目标路径
- 对于关键复制操作，建议实现客户端级别的重试机制
- 在日志中记录复制操作，便于后续审计和调试
