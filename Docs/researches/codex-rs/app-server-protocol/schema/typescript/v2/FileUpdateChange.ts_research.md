# FileUpdateChange.ts Research Document

## 场景与职责

`FileUpdateChange` 是 Codex App-Server 协议 v2 中用于描述文件更新变更的核心数据结构。它在文件补丁（Patch）应用流程中扮演关键角色，用于精确描述单个文件的变更内容，包括文件的新增、删除和修改操作。

主要应用场景包括：
- **代码审查**: 在 Agent 提出文件修改后，向用户展示详细的变更内容
- **补丁应用**: 作为 `patch/apply` 操作的核心数据结构，描述需要应用的变更
- **版本控制集成**: 与 Git 等版本控制系统交互，生成统一的 diff 格式
- **变更追踪**: 在会话历史中记录文件修改的详细信息

## 功能点目的

该类型的设计目标是提供一种标准化的方式来描述文件变更：

1. **精确描述变更**: 通过 `diff` 字段提供标准的统一 diff 格式，便于人类阅读和机器处理
2. **变更类型标识**: 通过 `kind` 字段明确标识变更是新增、删除还是更新
3. **路径追踪**: 支持文件重命名场景，通过 `move_path` 记录原路径
4. **多文件协调**: 作为数组元素，支持批量描述多个文件的变更

## 具体技术实现

### 数据结构定义

```typescript
// GENERATED CODE! DO NOT MODIFY BY HAND!
// This file was generated by [ts-rs](https://github.com/Aleph-Alpha/ts-rs).

import type { PatchChangeKind } from "./PatchChangeKind";

export type FileUpdateChange = { 
  path: string, 
  kind: PatchChangeKind, 
  diff: string, 
};
```

### 关键字段说明

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `path` | `string` | 是 | 文件的绝对路径或相对于工作区的路径，标识变更的目标文件 |
| `kind` | `PatchChangeKind` | 是 | 变更类型，标识是新增、删除还是更新操作 |
| `diff` | `string` | 是 | 统一 diff 格式的变更内容，描述文件的具体修改 |

#### PatchChangeKind 详解

```typescript
export type PatchChangeKind = 
  | { "type": "add" }           // 新增文件
  | { "type": "delete" }        // 删除文件
  | { "type": "update", move_path: string | null };  // 修改文件（可能包含重命名）
```

| 类型 | 说明 | diff 内容 |
|------|------|-----------|
| `add` | 新增文件 | 完整的新文件内容，通常以 `+` 开头 |
| `delete` | 删除文件 | 被删除文件的完整内容，通常以 `-` 开头 |
| `update` | 修改文件 | 标准的统一 diff 格式，显示行级变更 |

#### move_path 字段

当 `kind.type` 为 `"update"` 时，`move_path` 字段表示：
- `string`: 文件被重命名，此值为原文件路径
- `null`: 文件内容被修改，但路径未变更

## 风险、边界与改进建议

### 潜在风险

1. **diff 解析风险**: 非标准或格式错误的 diff 可能导致补丁应用失败
2. **路径安全问题**: `path` 字段需要严格验证，防止目录遍历攻击（如 `../../../etc/passwd`）
3. **大文件处理**: 大文件的 diff 可能非常庞大，影响传输和渲染性能
4. **编码问题**: diff 内容涉及多字节字符时，需要确保 UTF-8 编码一致性

### 边界情况

| 场景 | 处理建议 |
|------|----------|
| 空 diff | 对于 `add` 类型，表示创建空文件；对于 `update` 类型，可能是无效变更 |
| 二进制文件 | diff 格式不适用，需要特殊处理（如标记为二进制变更） |
| 权限变更 | 当前结构不记录文件权限变更，需要扩展 |
| 冲突解决 | 当目标文件在生成 diff 后被修改，需要冲突检测机制 |
| 循环重命名 | A→B 同时 B→A 的情况需要检测和阻止 |

### 改进建议

1. **添加元数据字段**:
   ```typescript
   export type FileUpdateChange = { 
     path: string, 
     kind: PatchChangeKind, 
     diff: string,
     metadata?: {
       originalSize?: number;    // 原始文件大小
       newSize?: number;         // 新文件大小
       lineCount?: number;       // 变更行数
       encoding?: string;        // 文件编码
     }
   };
   ```

2. **支持二进制文件**:
   ```typescript
   export type PatchChangeKind = 
     | { "type": "add", isBinary?: boolean }
     | { "type": "delete", isBinary?: boolean }
     | { "type": "update", move_path: string | null, isBinary?: boolean };
   ```

3. **添加变更统计**: 在 UI 展示时，添加变更统计信息（添加行数、删除行数）帮助用户快速评估变更规模

4. **增量 diff**: 对于超大文件，考虑支持增量 diff 或分块传输

5. **冲突标记**: 当检测到潜在冲突时，在 diff 中添加冲突标记（类似 Git 的冲突标记）

### 最佳实践

- 客户端应使用语法高亮渲染 diff，提升可读性
- 对于敏感文件（如配置文件、密钥文件），应添加额外的警告提示
- 建议实现折叠/展开功能，让用户可以专注于关心的变更部分
- 对于删除操作，应提供恢复机制或确认二次确认
