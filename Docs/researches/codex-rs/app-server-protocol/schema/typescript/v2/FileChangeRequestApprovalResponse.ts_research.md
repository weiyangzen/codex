# FileChangeRequestApprovalResponse.ts Research Document

## 场景与职责

`FileChangeRequestApprovalResponse` 是 Codex App-Server 协议 v2 中用于文件变更审批流程的核心响应类型。当 AI 代理（Agent）尝试对文件系统进行修改（如创建、更新、删除文件）时，系统会向用户展示变更请求，用户做出决策后，该类型用于封装用户的审批决定并返回给服务器。

此类型主要应用于以下场景：
- **文件修改审批**: 当 Agent 提出文件变更请求时，客户端需要向用户展示变更详情并收集用户决策
- **会话级权限管理**: 用户可以选择仅批准当前变更，或批准同一会话中对相同文件的后续所有变更
- **安全控制**: 作为沙箱安全策略的一部分，确保文件系统操作得到用户明确授权

## 功能点目的

该类型的核心目的是标准化文件变更审批响应的数据结构，确保：

1. **决策传递**: 将用户的审批决策（接受、拒绝、取消等）准确传递给服务器
2. **权限缓存控制**: 支持用户选择是否将审批决策缓存到会话级别，避免重复询问
3. **流程控制**: 通过 `cancel` 决策允许用户不仅拒绝变更，还立即中断当前 Agent 回合

该响应类型与 `FileChangeApprovalDecision` 枚举紧密配合，共同构成完整的文件变更审批机制。

|----|------|
| `"accept"` | 用户批准文件变更，Agent 继续执行 |

## 风险、边界与改进建议

### 潜在风险

1. **决策持久化风险**: `acceptForSession` 决策会将会话级权限缓存，如果用户忘记已授权的会话，可能导致意外的文件修改
2. **并发冲突**: 多个文件变更请求同时发出时，响应顺序可能与请求顺序不一致，客户端需要正确处理
3. **类型安全**: 作为自动生成的代码，手动修改可能导致与 Rust 端不一致

### 边界情况

| 场景 | 行为 |
|------|------|
| 空响应 | 服务器应拒绝处理，返回验证错误 |
| 未知决策值 | 反序列化失败，协议错误 |
| 网络超时 | 需要客户端重试机制 |
| 用户长时间未响应 | 服务器可能设置超时，自动拒绝 |

### 改进建议

1. **添加时间戳**: 考虑在响应中添加决策时间戳，便于审计和调试
2. **扩展决策理由**: 可考虑添加可选的 `reason` 字段，让用户说明拒绝原因
3. **批量审批**: 对于大量文件变更，考虑支持批量审批模式，减少用户交互次数
4. **决策撤销**: 考虑支持在极短时间内撤销已提交的决策
5. **版本控制**: 在响应中添加协议版本信息，便于未来演进

### 相关安全考虑

- 此类型涉及文件系统安全，建议配合完整的沙箱机制使用
- 客户端应验证服务器返回的变更详情与原始请求一致，防止中间人攻击
- 敏感操作的审批决策应考虑额外的身份验证步骤
