## 1. 场景与职责

### 1.1 核心定位

`v2.rs` 是 Codex App-Server Protocol 的 **v2 API 定义文件**，承担以下核心职责：

1. **协议类型定义**: 定义客户端与服务器之间 JSON-RPC 通信的所有数据结构
2. **核心/协议桥接**: 将 `codex-protocol` crate 的核心类型转换为 App-Server v2 API 类型
3. **实验性 API 标记**: 通过 `ExperimentalApi` trait 和宏标记不稳定接口
4. **TypeScript/JSON Schema 生成**: 支持生成客户端类型定义和 JSON Schema

### 1.2 使用场景

| 场景 | 说明 |
|------|------|
| **VSCode 扩展通信** | VSCode 插件通过 v2 协议与 App-Server 交互 |
| **CLI 工具集成** | Codex CLI 使用 v2 协议进行会话管理 |
| **第三方客户端** | 外部工具通过 WebSocket/Stdio 连接使用 v2 API |
| **内部服务通信** | Codex Cloud 等内部服务使用 v2 协议子集 |

### 1.3 与 v1 协议的关系

- v1 协议已冻结，仅保留向后兼容的 deprecated API
- **所有新功能开发必须在 v2 中进行**（根据 AGENTS.md 规范）
- v2 提供更细粒度的线程/回合/项目生命周期管理

## 2. 功能点目的

### 2.1 主要功能模块

```
v2.rs 功能结构:
├── 错误类型 (CodexErrorInfo)
├── 配置相关类型 (Config, ProfileV2, ConfigLayerSource 等)
├── 审批策略 (AskForApproval, ApprovalsReviewer)
├── 沙箱策略 (SandboxMode, SandboxPolicy)
├── 账户管理 (Account, LoginAccountParams/Response 等)
├── 模型管理 (ModelListParams/Response, Model 等)
├── 线程生命周期 (ThreadStart/Resume/Fork/Archive 等)
├── 回合管理 (TurnStart/Steer/Interrupt)
├── 文件系统 API (FsReadFile/FsWriteFile/...)
├── 命令执行 (CommandExecParams/Response/...)
├── MCP 服务器 (McpServerStatus, McpServerElicitationRequest 等)
├── 技能与插件 (SkillsList, PluginList/Install/Uninstall)
├── 实时会话 (ThreadRealtimeStart/AppendAudio/...)
├── 通知类型 (ThreadStartedNotification, TurnCompletedNotification 等)
└── 审批请求 (CommandExecutionRequestApproval, FileChangeRequestApproval)
```

### 2.2 关键功能详解

#### 2.2.1 线程生命周期管理

| 方法 | 用途 |
|------|------|
| `thread/start` | 创建新线程，支持配置覆盖 |
| `thread/resume` | 从磁盘或内存恢复线程 |
| `thread/fork` | 从现有线程分叉创建新线程 |
| `thread/archive` | 归档线程 |
| `thread/unarchive` | 取消归档 |
| `thread/rollback` | 回滚指定数量的回合 |
| `thread/read` | 读取线程详情（可选包含回合） |
| `thread/list` | 分页列出线程 |

#### 2.2.2 回合管理

| 方法 | 用途 |
|------|------|
| `turn/start` | 启动新回合，提交用户输入 |
| `turn/steer` | 在活跃回合中追加输入 |
| `turn/interrupt` | 中断当前回合 |

#### 2.2.3 审批系统

- **命令执行审批**: `CommandExecutionRequestApprovalParams/Response`
- **文件变更审批**: `FileChangeRequestApprovalParams/Response`
- **权限请求审批**: `PermissionsRequestApprovalParams/Response`
- **MCP 引发审批**: `McpServerElicitationRequestParams/Response`

#### 2.2.4 实验性功能标记

使用 `#[experimental("reason")]` 属性标记不稳定 API：

```rust
#[experimental("askForApproval.granular")]
Granular { ... }

#[experimental("thread/start.dynamicTools")]
dynamic_tools: Option<Vec<DynamicToolSpec>>
```

## 3. 具体技术实现

### 3.1 核心宏

#### 3.1.1 `v2_enum_from_core!`

用于从 `codex-protocol` 核心枚举快速生成 v2 API 枚举：

```rust
macro_rules! v2_enum_from_core {
    (
        $(#[$enum_meta:meta])*
        pub enum $Name:ident from $Src:path {
            $( $(#[$variant_meta:meta])* $Variant:ident ),+ $(,)?
        }
    ) => { ... }
}
```

**生成的内容**:
- 带 `#[serde(rename_all = "camelCase")]` 的枚举
- `#[ts(export_to = "v2/")]` TypeScript 导出配置
- `to_core()` 方法：v2 → core 转换
- `From<$Src>` 实现：core → v2 转换

**使用示例**:
```rust
v2_enum_from_core!(
    pub enum HookEventName from CoreHookEventName {
        SessionStart, UserPromptSubmit, Stop
    }
);
```

### 3.2 序列化/反序列化模式

#### 3.2.1 Double Option 处理

用于区分 "未设置" 和 "显式设置为 null"：

```rust
#[serde(
    default,
    deserialize_with = "super::serde_helpers::deserialize_double_option",
    serialize_with = "super::serde_helpers::serialize_double_option",
    skip_serializing_if = "Option::is_none"
)]
#[ts(optional = nullable)]
pub service_tier: Option<Option<ServiceTier>>,
```

#### 3.2.2 带 Tag 的枚举

用于类型安全的 discriminated unions：

```rust
#[derive(Serialize, Deserialize, Debug, Clone, PartialEq, JsonSchema, TS)]
#[serde(tag = "type", rename_all = "camelCase")]
#[ts(tag = "type")]
#[ts(export_to = "v2/")]
pub enum ThreadItem {
    #[serde(rename_all = "camelCase")]
    #[ts(rename_all = "camelCase")]
    UserMessage { id: String, content: Vec<UserInput> },
    // ...
}
```

### 3.4 配置层系统

`ConfigLayerSource` 定义配置优先级（数字越小优先级越低）：

## 6. 风险、边界与改进建议

### 6.1 已知风险

#### 6.1.1 实验性 API 稳定性

- 大量 API 标记为 `#[experimental]`，可能在未来版本中变更
- 客户端必须处理实验性字段可能被剥离的情况（如 `strip_experimental_fields` 方法）

#### 6.1.2 类型转换开销

- Core 类型与 V2 类型之间的转换存在运行时开销
- 复杂嵌套结构（如 `ThreadItem`）的转换逻辑复杂，容易出错

#### 6.1.3 序列化兼容性

- `#[serde(deny_unknown_fields)]` 在某些类型上使用，可能导致前向兼容性问题
- Double option 模式使用复杂，容易误用

### 6.2 边界情况

#### 6.2.1 路径处理

- 使用 `AbsolutePathBuf` 强制要求绝对路径
- Windows/Unix 路径格式差异需要特别处理

#### 6.2.2 空值处理

```rust
// 区分 "未设置" vs "显式 null"
pub service_tier: Option<Option<ServiceTier>>

// 序列化行为:
// - None: 字段被省略
// - Some(None): 字段为 null
// - Some(Some(tier)): 字段为 tier 值
```

#### 6.2.3 大列表分页

- `ThreadListParams`, `ModelListParams` 等使用游标分页
- 客户端需要处理 `next_cursor` 为 `None` 表示无更多数据

### 6.3 改进建议

#### 6.3.1 代码组织

1. **模块拆分**: 当前文件接近 8000 行，建议按功能拆分为子模块：
   ```
   protocol/v2/
   ├── mod.rs          # 公共导出
   ├── config.rs       # 配置相关类型
   ├── thread.rs       # 线程生命周期
   ├── turn.rs         # 回合管理
   ├── items.rs        # ThreadItem 类型
   ├── approval.rs     # 审批相关
   ├── fs.rs           # 文件系统 API
   ├── command.rs      # 命令执行
   └── notification.rs # 通知类型
   ```

#### 6.3.2 类型安全

2. **新类型模式**: 对 ID 类型使用新类型包装：
   ```rust
   pub struct ThreadId(String);
   pub struct TurnId(String);
   ```

3. **验证器**: 在反序列化时添加更多验证：
   ```rust
   #[serde(deserialize_with = "validate_thread_id")]
   pub thread_id: String,
   ```

#### 6.3.3 文档

4. **字段文档**: 为复杂字段添加更多示例：
   ```rust
   /// 沙箱策略配置
   /// 
   /// # 示例
   /// ```json
   /// { "type": "readOnly", "access": { "type": "fullAccess" } }
   /// ```
   pub sandbox_policy: Option<SandboxPolicy>,
   ```

#### 6.3.4 测试

5. **属性测试**: 对序列化/反序列化使用 `proptest` 进行模糊测试
6. **兼容性测试**: 添加 v2 与 core 类型转换的 round-trip 测试

#### 6.3.5 性能

7. **零拷贝反序列化**: 对大型二进制数据（如文件内容）考虑使用 `&[u8]` 而非 `String`
8. **延迟序列化**: 对不常用字段使用 `#[serde(skip_serializing_if = "Option::is_none")]`
