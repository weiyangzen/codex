## 1. 场景与职责

### 1.1 核心定位
`v1.rs` 是 Codex App-Server Protocol 的 **V1 版本 API 类型定义模块**，定义了早期（现已标记为 DEPRECATED）的 JSON-RPC API 请求/响应数据结构。该模块属于 `app-server-protocol` crate，负责客户端与 App-Server 之间的通信契约。

### 1.2 历史演进背景
- **V1 API**：早期设计，使用驼峰命名（camelCase）的扁平化 API，主要用于 VSCode 扩展等早期客户端
- **V2 API**：当前主推版本（`v2.rs`），使用 `resource/method` 格式的命名空间（如 `thread/start`），功能更丰富
- **现状**：V1 API 已冻结，新功能仅在 V2 中开发，但 V1 类型仍被保留用于向后兼容

### 1.3 使用场景
| 场景 | 说明 |
|------|------|
| 向后兼容 | 旧版客户端（如早期 VSCode 扩展）仍依赖 V1 类型 |
| 类型桥接 | 部分 V1 类型被 V2 复用或映射（如 `InitializeParams`） |
| 遗留功能 | 如 `GetConversationSummary`、`GitDiffToRemote` 等功能仅在 V1 中定义 |

### 2.2 各功能点详细说明

#### 2.2.1 初始化握手（仍在使用）

## 3. 具体技术实现

### 3.1 序列化/反序列化配置

所有类型统一使用以下 derive 宏配置：

```rust
#[derive(Serialize, Deserialize, Debug, Clone, PartialEq, JsonSchema, TS)]
#[serde(rename_all = "camelCase")]  // 字段名驼峰命名
```

| 属性 | 用途 |
|------|------|
| `JsonSchema` | 生成 JSON Schema 供文档和验证 |
| `TS` (ts_rs) | 生成 TypeScript 类型定义 |
| `camelCase` | 与 JSON-RPC 惯例保持一致 |

### 3.2 特殊序列化处理

#### Untagged Enum（GetConversationSummaryParams）
```rust
#[serde(untagged)]
pub enum GetConversationSummaryParams {
    RolloutPath { rollout_path: PathBuf },
    ThreadId { conversation_id: ThreadId },
}
```
- 根据字段名自动区分变体，无需显式类型标签

#### 条件跳过序列化
```rust
#[serde(skip_serializing_if = "Option::is_none")]
pub capabilities: Option<InitializeCapabilities>,
```
- `None` 值字段在序列化时被省略，减少传输体积

### 3.3 类型依赖关系


## 6. 风险、边界与改进建议

### 6.1 当前风险

| 风险点 | 严重度 | 说明 |
|--------|--------|------|
| 技术债务 | 中 | V1 类型与 V2 类型并存，维护成本增加 |
| 命名不一致 | 低 | V1 使用驼峰命名，部分 V2 使用 snake_case（如 config） |
| 实验性标记缺失 | 低 | V1 类型无 `#[experimental]` 标记机制（V2 有） |
| 测试覆盖 | 低 | V1 专属功能的测试主要依赖集成测试 |

### 6.2 边界情况

1. **Initialize 是 V1 中唯一非废弃的核心功能**
   - 所有客户端连接必须通过 `initialize` 握手
   - `InitializeCapabilities` 的 `opt_out_notification_methods` 是连接级通知过滤机制

2. **GetConversationSummary 的双模式查询**
   - `RolloutPath` 变体支持相对路径，会从 `CODEX_HOME` 解析
   - `ThreadId` 变体需要有效的 UUID 格式

3. **审批参数的兼容性**
   - V1 的 `ApplyPatchApprovalParams` 和 `ExecCommandApprovalParams` 仍被旧版 `ServerRequest` 使用
   - 新版代码应使用 V2 的 `CommandExecutionRequestApprovalParams` 和 `FileChangeRequestApprovalParams`

### 6.3 改进建议

#### 短期（维护性）
1. **文档补充**：为 V1 废弃类型添加更详细的废弃说明，指向 V2 替代方案
   ```rust
   /// DEPRECATED: Use `v2::CommandExecutionRequestApprovalParams` instead.
   /// This type is kept for backward compatibility with legacy clients.
   ```

2. **测试迁移**：将依赖 V1 废弃 API 的测试逐步迁移到 V2 等效 API

#### 中期（架构优化）
3. **类型隔离**：考虑将 V1 废弃类型移至 `protocol/v1/deprecated.rs` 子模块，减少主命名空间污染

4. **移除计划**：制定 V1 API 的正式移除时间表，通知客户端迁移

#### 长期（技术债务清理）
5. **统一配置系统**：`UserSavedConfig` 与 V2 `Config` 存在功能重叠，考虑统一或明确分层

6. **移除遗留类型**：当所有已知客户端迁移完成后，移除以下类型：
   - `GetConversationSummaryParams/Response`
   - `GitDiffToRemoteParams/Response`
   - `GetAuthStatusParams/Response`
   - `LoginApiKeyParams`

### 6.4 相关文件修改建议

| 文件 | 建议 |
|------|------|
| `v1.rs` | 为每个废弃类型添加 `#[deprecated]` 属性（当前尚未添加） |
| `common.rs` | 在 `ClientRequest` 的 V1 变体上添加废弃文档 |
| `lib.rs` | 考虑将 V1 导出组织为 `pub mod v1 { ... }` 命名空间 |
| `README.md` | 更新文档，明确标记 V1 API 为废弃状态 |
