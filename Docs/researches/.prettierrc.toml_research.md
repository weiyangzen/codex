# .prettierrc.toml 研究文档

## 场景与职责

`.prettierrc.toml` 是 **Prettier** 代码格式化工具的主配置文件，用于定义代码格式化的规则和风格。该文件位于项目根目录，确保所有 JavaScript、TypeScript、JSON、Markdown 等文件在整个项目中保持一致的格式风格。

Prettier 是一个固执己见的（opinionated）代码格式化工具，通过减少可配置的选项来避免团队内部关于代码风格的争论。`.prettierrc.toml` 中的配置反映了 Codex 项目的代码风格偏好。

## 功能点目的

### 1. 打印宽度

```toml
printWidth = 80
```

**目的**：设置每行的最大字符数。

**技术细节**：
- 默认值为 80
- 当代码超过此宽度时，Prettier 会尝试换行
- 影响可读性和代码审查体验

**选择 80 的原因**：
- 传统终端宽度
- 方便并排查看两个文件
- 与许多现有工具和标准一致

**对比**：
- `.markdownlint-cli2.yaml` 中设置 `line_length: 100`
- 这意味着 Markdown 文件可以更长，而代码保持紧凑

### 2. 对象属性引号

```toml
quoteProps = "consistent"
```

**目的**：统一对象属性引号的使用。

**可选值**：
- `"as-needed"`（默认）：仅在需要时添加引号
- `"consistent"`：如果至少有一个属性需要引号，则全部添加
- `"preserve"`：保持输入的引号风格

**选择 "consistent" 的原因**：
- 保持代码风格一致性
- 避免混合使用带引号和不带引号的属性
- 提高可读性

**示例**：
```javascript
// consistent
{
  "foo-bar": 1,
  "baz": 2  // 因为 foo-bar 需要引号，baz 也加上
}

// as-needed
{
  "foo-bar": 1,
  baz: 2  // 不需要引号
}
```

### 3. 分号

```toml
semi = true
```

**目的**：在语句末尾添加分号。

**技术背景**：
- JavaScript 有自动分号插入（ASI）机制
- 但 ASI 有一些边界情况可能导致意外行为

**选择 `true` 的原因**：
- 明确标记语句结束
- 避免 ASI 的潜在陷阱
- 与大多数 JavaScript 代码库一致

**示例**：
```javascript
// semi: true
const x = 1;
const y = 2;

// semi: false
const x = 1
const y = 2
```

### 4. 缩进宽度

```toml
tabWidth = 2
```

**目的**：设置缩进的空格数。

**选择 2 的原因**：
- 现代 JavaScript/TypeScript 项目的标准
- 在深度嵌套时保持代码紧凑
- 与大多数开源项目一致

**对比**：
- Python 社区通常使用 4 个空格
- Go 使用制表符（但显示为 8 个空格）

### 5. 尾随逗号

```toml
trailingComma = "all"
```

**目的**：在多行结构中尽可能添加尾随逗号。

**可选值**：
- `"none"`：不添加尾随逗号
- `"es5"`：在 ES5 有效的位置添加（对象、数组）
- `"all"`：在所有可能的位置添加（包括函数参数）

**选择 "all" 的原因**：
- 减少 git diff（添加新行时只显示一行变更）
- 方便复制粘贴代码行
- 与现代 JavaScript 引擎兼容

**示例**：
```javascript
// trailingComma: "all"
const obj = {
  a: 1,
  b: 2,  // 尾随逗号
};

function foo(
  a,
  b,  // 尾随逗号
) {}
```

### 6. Markdown 文本换行

```toml
# Preserve existing behavior for markdown/text wrapping.
proseWrap = "preserve"
```

**目的**：控制 Markdown 中文本的换行行为。

**可选值**：
- `"always"`：始终换行到 printWidth
- `"never"`：不自动换行
- `"preserve"`：保持输入的换行（默认）

**选择 "preserve" 的原因**：
- 注释说明："Preserve existing behavior"
- 避免意外改变 Markdown 文档的结构
- 与 `.markdownlint-cli2.yaml` 的 `line_length: 100` 协调

**影响**：
- Prettier 不会重新格式化 Markdown 段落
- 手动换行会被保留

## 具体技术实现

### 文件格式

`.prettierrc.toml` 使用 TOML 格式：

```toml
key = value
key2 = "string value"
key3 = true
```

TOML 特点：
- 人类可读
- 明确的类型
- 支持注释

### 配置优先级

```
1. 命令行参数（--print-width 80）
2. 项目根目录的 .prettierrc.toml
3. 上级目录的 .prettierrc.toml
4. 其他支持的文件名（.prettierrc, prettier.config.js 等）
5. package.json 中的 "prettier" 字段
6. Prettier 默认值
```

### 支持的文件类型

Prettier 可以格式化：
- JavaScript/TypeScript (`.js`, `.ts`, `.jsx`, `.tsx`)
- JSON (`.json`)
- Markdown (`.md`)
- YAML (`.yml`, `.yaml`)
- HTML (`.html`)
- CSS/SCSS/Less
- GraphQL
- 等等

## 关键代码路径与文件引用

### 相关文件

1. **.prettierignore**
   - 定义哪些文件不应该被格式化
   - 与 `.prettierrc.toml` 配合使用

2. **package.json**
   - 包含 Prettier 的 npm 脚本
   ```json
   "scripts": {
     "format": "prettier --check *.json *.md docs/*.md .github/workflows/*.yml **/*.js",
     "format:fix": "prettier --write *.json *.md docs/*.md .github/workflows/*.yml **/*.js"
   }
   ```

3. **.markdownlint-cli2.yaml**
   - Markdown linting 配置
   - `line_length: 100` 与 Prettier 的 `printWidth: 80` 需要协调

4. **.editorconfig**（如果存在）
   - 编辑器配置
   - Prettier 会读取 `indent_size` 和 `max_line_length`

### 格式化流程

```
1. Prettier 接收输入文件
2. 解析文件类型
3. 读取 .prettierrc.toml 配置
4. 应用相应解析器和格式化规则
5. 输出格式化后的代码
```

## 依赖与外部交互

### Prettier 工具

**安装方式**：
```bash
# npm
npm install --save-dev prettier

# pnpm
pnpm add -D prettier
```

**版本**：
```json
// package.json
"devDependencies": {
  "prettier": "^3.5.3"
}
```

**基本使用**：
```bash
# 检查格式
prettier --check .

# 修复格式
prettier --write .

# 格式化特定文件
prettier --write src/**/*.ts

# 使用特定配置
prettier --config .prettierrc.toml --write .
```

### 与 ESLint 的集成

Prettier 和 ESLint 可能有冲突的规则。解决方案：

```bash
# 安装 eslint-config-prettier
pnpm add -D eslint-config-prettier

# .eslintrc.js
module.exports = {
  extends: [
    'eslint:recommended',
    'prettier',  // 禁用与 Prettier 冲突的规则
  ],
};
```

### CI/CD 集成

```yaml
# .github/workflows/format.yml
name: Format Check
on: [push, pull_request]
jobs:
  format:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - run: pnpm install
      - run: pnpm run format
```

### 编辑器集成

**VS Code**：
- 扩展：esbenp.prettier-vscode
- 设置：
  ```json
  {
    "editor.defaultFormatter": "esbenp.prettier-vscode",
    "editor.formatOnSave": true
  }
  ```

**Vim**：
- 插件：ALE 或 coc-prettier
- 配置自动格式化

## 风险、边界与改进建议

### 潜在风险

1. **与 markdownlint 的行长度冲突**
   - Prettier: `printWidth = 80`
   - markdownlint: `line_length = 100`
   - 可能导致 Markdown 文件格式不一致

2. **proseWrap = "preserve" 的副作用**
   - 不会自动格式化长段落
   - 需要手动维护 Markdown 换行

3. **版本兼容性**
   - Prettier 3.x 与 2.x 有一些行为差异
   - 升级时可能需要调整配置

### 边界情况

1. **生成的代码**
   - OpenAPI 生成的代码可能不符合 Prettier 风格
   - 需要在 `.prettierignore` 中排除

2. **第三方代码**
   - `vendor` 目录或复制的代码
   - 应该被排除

3. **模板字符串**
   - Prettier 可能重新格式化模板字符串
   - 可能影响输出

### 改进建议

1. **协调 Markdown 配置**
   ```toml
   # 选项 1：让 Prettier 处理 Markdown 换行
   proseWrap = "always"
   printWidth = 100  # 与 markdownlint 一致
   
   # 选项 2：完全禁用 Prettier 的 Markdown 格式化
   # 在 .prettierignore 中添加 *.md
   ```

2. **添加更多配置项**
   ```toml
   # 箭头函数参数括号
   arrowParens = "always"
   
   # 对象花括号内空格
   bracketSpacing = true
   
   # JSX 闭合标签位置
   jsxBracketSameLine = false
   
   # 单引号
   singleQuote = true
   ```

3. **添加 overrides**
   ```toml
   [[overrides]]
   files = "*.json"
   options = {
     printWidth = 120  # JSON 可以更宽
   }
   
   [[overrides]]
   files = "*.md"
   options = {
     proseWrap = "never"  # Markdown 不自动换行
   }
   ```

4. **添加注释说明**
   ```toml
   # Prettier configuration for Codex
   # See: https://prettier.io/docs/en/options.html
   
   # Line length (consistent with common standards)
   printWidth = 80
   
   # Quote style for object properties
   quoteProps = "consistent"
   
   # Semicolons for statement termination
   semi = true
   
   # Indentation (2 spaces is standard for JS/TS)
   tabWidth = 2
   
   # Trailing commas (reduces git diff noise)
   trailingComma = "all"
   
   # Markdown text wrapping (preserve manual formatting)
   proseWrap = "preserve"
   ```

5. **考虑 EditorConfig 集成**
   ```ini
   # .editorconfig
   root = true
   
   [*]
   indent_style = space
   indent_size = 2
   max_line_length = 80
   
   [*.md]
   max_line_length = 100
   ```

### 使用示例

```bash
# 检查格式问题
npx prettier --check .

# 修复格式问题
npx prettier --write .

# 检查特定文件
npx prettier --check src/index.ts

# 格式化并输出到 stdout
npx prettier --stdin-filepath file.ts < file.ts

# 查看配置
npx prettier --find-config-path .
npx prettier --config-precedence prefer-file --stdin-filepath file.ts < file.ts
```

### 配置对比

| 项目 | 当前配置 | 推荐配置 |
|------|---------|---------|
| printWidth | 80 | 80（保持一致） |
| tabWidth | 2 | 2（保持一致） |
| semi | true | true（保持一致） |
| singleQuote | 未设置（false） | 考虑设为 true |
| trailingComma | "all" | "all”（保持一致） |
| proseWrap | "preserve" | 考虑 "always" 或根据文件类型覆盖 |

**JavaScript/TypeScript 社区趋势**：
- 越来越多的项目使用单引号（`singleQuote = true`）
- 但双引号在 JSON 和 HTML 属性中更常见
- 当前配置使用双引号是合理的选择
