# codex-cli/.gitignore 研究文档

## 场景与职责

`.gitignore` 文件位于 `codex-cli/` 目录下，用于指定 Git 版本控制应忽略的文件和目录。这是防止构建产物、依赖目录和敏感文件被意外提交到代码仓库的标准实践。

该文件专门针对 `codex-cli` 子项目，与仓库根目录的 `.gitignore` 形成层级化的忽略规则体系。

## 功能点目的

### 核心功能
- **排除 `/vendor/` 目录**：防止本地安装的依赖和二进制文件被提交到 Git

### 目的详解

1. **版本控制纯净性**
   - `vendor/` 目录包含通过 `install_native_deps.py` 下载的原生二进制文件（如 Rust CLI、ripgrep）
   - 这些文件体积大、平台相关，不适合版本控制

2. **跨平台兼容性**
   - 不同平台（Linux x64/arm64、macOS x64/arm64、Windows）需要不同的二进制文件
   - 提交特定平台的二进制文件会导致其他平台开发者的工作目录污染

3. **构建可复现性**
   - 通过脚本重新下载/生成依赖，确保所有开发者从一致的状态开始
   - 避免"已编译二进制文件与源代码不同步"的问题

## 具体技术实现

### 文件内容
```
/vendor/
```

### 技术细节
- **语法**：Git ignore 模式的简化 glob 语法
- **前导斜杠 `/`**：表示从 `.gitignore` 所在目录开始匹配（即仅匹配 `codex-cli/vendor/`，不匹配子目录中的 `vendor/`）
- **后导斜杠 `/`**：表示仅匹配目录，不匹配同名文件

### 与安装脚本的协作

在 `codex-cli/scripts/install_native_deps.py` 中：
```python
VENDOR_DIR_NAME = "vendor"
vendor_dir = codex_cli_root / VENDOR_DIR_NAME
vendor_dir.mkdir(parents=True, exist_ok=True)
```

该脚本将下载的二进制文件安装到 `vendor/` 目录，而 `.gitignore` 确保这些文件不会被提交。

### 目录结构示意
```
codex-cli/
├── .gitignore          # 排除 /vendor/
├── bin/
│   ├── codex.js        # Node.js 入口脚本
│   └── rg              # DotSlash 清单（用于下载 ripgrep）
├── scripts/
│   ├── install_native_deps.py  # 安装脚本，生成 vendor/
│   └── build_npm_package.py    # 打包脚本，读取 vendor/
└── vendor/             # 被忽略，包含：
    ├── x86_64-unknown-linux-musl/
    │   ├── codex/
    │   └── path/
    │       └── rg
    └── ... (其他平台)
```

## 关键代码路径与文件引用

### 直接关联文件
| 文件 | 关系 | 说明 |
|------|------|------|
| `codex-cli/scripts/install_native_deps.py` | 生成者 | 下载并安装二进制到 `vendor/` |
| `codex-cli/scripts/build_npm_package.py` | 消费者 | 从 `vendor/` 复制二进制到发布包 |
| `codex-cli/bin/codex.js` | 运行时消费者 | 从 `vendor/` 加载原生二进制 |

### 数据流
```
GitHub Actions Artifacts
    ↓ (gh run download)
install_native_deps.py
    ↓ (解压到)
vendor/<target>/<component>/
    ↓ (被 .gitignore 排除，不进入 Git)
build_npm_package.py
    ↓ (复制到)
staging/vendor/
    ↓ (npm pack)
@openai/codex-<platform>.tgz
```

## 依赖与外部交互

### 上游依赖
- Git：解析 `.gitignore` 规则
- `install_native_deps.py`：填充 `vendor/` 目录

### 下游影响
- Git 工作目录：`vendor/` 中的变更不会显示为未跟踪文件
- npm 发布流程：`vendor/` 内容通过 `build_npm_package.py` 复制到发布包

### 与根目录 .gitignore 的关系
仓库根目录的 `.gitignore` 包含更通用的规则：
```
node_modules/
.DS_Store
*.log
...
```

`codex-cli/.gitignore` 专门针对 CLI 子项目的特定需求（原生二进制文件管理）。

## 风险、边界与改进建议

### 当前风险
1. **规则过于简单**：仅排除 `/vendor/`，未考虑其他可能生成的文件
2. **平台特定文件**：如果开发者在其他位置生成平台特定文件，可能被意外提交

### 边界情况
- 前导 `/` 确保只匹配当前目录的 `vendor/`，但如果子目录也有 `vendor/` 需求，需要额外规则
- 如果 `vendor/` 需要部分提交（如某些通用配置文件），当前规则会全部排除

### 改进建议

1. **扩展排除规则**（推荐）
   ```gitignore
   # Native binaries installed by install_native_deps.py
   /vendor/
   
   # Build outputs
   /dist/
   /build/
   
   # Logs
   *.log
   npm-debug.log*
   
   # Environment files
   .env
   .env.local
   ```

2. **添加注释说明**
   ```gitignore
   # Native binaries downloaded by scripts/install_native_deps.py
   # These are platform-specific and should not be committed.
   /vendor/
   ```

3. **考虑添加 dist/ 排除**
   - `build_container.sh` 生成 `dist/codex.tgz`
   - 如果该目录也用于其他构建产物，应考虑排除

### 一致性检查
- 当前 `codex-cli/.dockerignore` 仅排除 `node_modules/`
- 如果 Docker 构建也会生成或使用 `vendor/`，应同步更新 `.dockerignore`
