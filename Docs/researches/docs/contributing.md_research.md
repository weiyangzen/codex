# contributing.md 研究文档

## 场景与职责

contributing.md 是 Codex CLI 项目的贡献指南文档，明确了外部贡献的政策、流程和规范。该文档特别强调了**仅接受邀请的外部贡献**政策，并详细说明了受邀贡献者的工作流程。

**适用场景：**
- 外部开发者希望为项目做贡献
- 项目维护者处理外部贡献
- 受邀贡献者了解开发工作流程

## 功能点目的

### 1. 外部贡献政策
- **核心政策**：仅接受邀请的外部贡献
- **原因**：
  - 过去的开源贡献模式难以扩展
  - 许多贡献缺乏架构上下文、系统级约束或路线图考虑
  - 审查和迭代 PR 往往比直接实现修复花费更多时间
  - 最有价值的贡献来自对问题领域有深入理解的社区成员

### 2. 贡献邀请条件
项目团队可能在以下情况下邀请外部贡献：
- 问题已被充分理解
- 提议的方法与团队预期的解决方案一致
- 问题被认为是高影响和高优先级的

### 3. 开发工作流程
**分支管理**：
- 从 `main` 创建主题分支（如 `feat/interactive-prompt`）
- 保持变更聚焦，不相关的修复应作为单独的 PR

**代码质量**：
- 确保变更无 lint 警告和测试失败

### 4. 受邀代码贡献指导

**步骤**：
1. **从 issue 开始** - 开新 issue 或在现有讨论中评论，在编写代码前达成共识
2. **添加或更新测试** - 错误修复应附带测试覆盖，100% 覆盖不是必需的，但要有有意义的断言
3. **记录行为** - 如果变更影响用户可见的行为，更新 README、内联帮助或相关示例项目
4. **保持提交原子性** - 每个提交应该能够编译并通过测试

### 5. 模型元数据更新
当更新模型目录或模型元数据时：
- 为不支持图像的模型显式设置 `input_modalities`
- 记住兼容性默认值：省略 `input_modalities` 目前意味着支持文本 + 图像
- 确保接受图像的客户端界面使用相同的能力信号
- 添加/更新覆盖不支持图像行为和警告路径的测试

### 6. 提交 PR 流程
**要求**：
- 填写 PR 模板（或包含类似信息）- **What? Why? How?**
- 包含指向 issue 跟踪器中错误报告或增强请求的链接
- 在本地运行**所有**检查，使用根 `just` 辅助工具
- 确保分支与 `main` 保持最新并解决合并冲突
- 仅在 PR 处于可合并状态时才标记为**Ready for review**

**本地检查命令**：
```bash
just fmt
just fix -p <crate>
cargo test -p codex-tui
just test  # 如果需要完整扫描
```

### 7. 审查流程
1. 一名维护者将被分配为主要审查者
2. 如果受邀 PR 引入了之前未讨论和批准的范围或行为，可能会被关闭
3. 可能会要求更改，请不要个人化
4. 当达成共识认为 PR 达到标准时，维护者将进行 squash-and-merge

### 8. 社区价值观
- **友善和包容** - 遵循 [Contributor Covenant](https://www.contributor-covenant.org/)
- **假设善意** - 书面沟通很难，倾向于宽容
- **教学相长** - 如果发现令人困惑的地方，开 issue 或讨论提供建议

### 9. 获取帮助
- 在 Discussion 主题中提问或加入相关 issue

### 10. CLA 签署
**流程**：
1. 打开 Pull Request
2. 粘贴以下评论（或如果之前签署过则回复 `recheck`）：
   ```
   I have read the CLA Document and I hereby sign the CLA
   ```
3. CLA-Assistant bot 在仓库中记录签名并标记状态检查通过

### 11. 安全与负责任 AI
- 发现漏洞或对模型输出有疑虑？发送邮件至 **security@openai.com**

## 具体技术实现

### PR 工作流程

```
发现/报告问题
    ↓
等待团队邀请（如果被邀请）
    ↓
创建主题分支
    ↓
开发变更（保持原子提交）
    ↓
本地运行检查
    - just fmt
    - just fix -p <crate>
    - cargo test -p <crate>
    ↓
提交 PR（填写模板）
    ↓
签署 CLA
    ↓
审查迭代
    ↓
Squash-and-merge
```

### 本地开发检查清单

```bash
# 格式化
just fmt

# 修复指定 crate 的 lint
just fix -p <crate-you-touched>

# 运行相关测试（项目特定最快）
cargo test -p codex-tui

# 如果需要完整扫描
just test

# 避免使用 --all-features 进行常规本地运行
```

## 风险、边界与改进建议

### 潜在风险

1. **贡献者挫败感**
   - 严格的邀请制政策可能让潜在贡献者感到沮丧
   - 建议：明确说明哪些类型的贡献更可能被邀请

2. **知识流失**
   - 限制外部贡献可能导致社区知识积累不足
   - 建议：增加文档和教程贡献的机会

3. **审查瓶颈**
   - 即使是受邀贡献也可能面临审查延迟
   - 建议：明确审查时间预期

### 边界情况

1. **文档贡献**
   - 文档是否也需要邀请？

2. **小修复**
   - 拼写错误、文档链接修复等微小变更

3. **安全相关贡献**
   - 安全漏洞报告的处理流程

### 改进建议

1. **贡献类型明确化**
   - 创建贡献类型矩阵，明确哪些需要邀请
   - 示例：
     - 需要邀请：功能添加、架构变更
     - 可能不需要：文档修复、拼写错误

2. **快速通道**
   - 为微小修复建立快速审查通道

3. **社区参与**
   - 定期举办社区办公时间
   - 提供设计讨论的早期参与机会

4. **透明度提升**
   - 公开路线图，让社区了解优先事项
   - 定期发布贡献统计和致谢

5. **导师计划**
   - 为新贡献者分配导师
   - 提供结对编程机会
