# .npmrc 研究文档

## 场景与职责

`.npmrc` 是 npm/pnpm 包管理器的配置文件，用于定义包安装行为、依赖解析策略和注册表设置。该文件位于项目根目录，确保所有开发者和 CI 环境使用一致的包管理配置。

Codex 项目使用 **pnpm** 作为主要的 Node.js 包管理器，`.npmrc` 中的配置针对 pnpm 的特定功能进行了优化。

## 功能点目的

### 1. 依赖提升（Shamefully Hoist）

```ini
shamefully-hoist=true
```

**目的**：允许将依赖提升到 `node_modules` 根目录。

**技术背景**：
- pnpm 使用内容寻址存储和硬链接，默认创建严格的依赖树
- 默认情况下，依赖只能访问其 `package.json` 中声明的依赖
- `shamefully-hoist=true` 模拟 npm/yarn 的行为，允许未声明的依赖访问

**使用场景**：
- 某些遗留工具或库可能依赖未显式声明的 peer dependencies
- 与不完全兼容 pnpm 严格模式的工具集成
- 迁移自 npm/yarn 的项目过渡期

**风险**：
- 可能隐藏缺失依赖声明的问题
- 导致"在我的机器上能运行"的问题

### 2. 严格对等依赖检查

```ini
strict-peer-dependencies=false
```

**目的**：禁用严格的对等依赖检查。

**技术背景**：
- pnpm 默认严格检查 peer dependencies
- 如果 peer dependency 未满足，安装会失败
- 设置为 `false` 时，仅发出警告而不阻止安装

**使用场景**：
- 某些包可能声明了不必要或冲突的 peer dependencies
- 快速原型开发时暂时忽略 peer dependency 问题

**建议**：
- 长期项目应考虑修复 peer dependency 问题，而非禁用检查
- 可以在 CI 中启用严格检查

### 3. 节点链接器模式

```ini
node-linker=hoisted
```

**目的**：使用提升的节点链接器模式。

**技术背景**：
- pnpm 支持多种 node_modules 布局策略：
  - `isolated`（默认）：严格隔离，使用符号链接
  - `hoisted`：提升依赖到根目录
  - `pnp`：Plug'n'Play 模式（类似 Yarn 2+）

**`hoisted` 模式特点**：
- 创建扁平的 `node_modules` 结构
- 与 npm/yarn 的行为更相似
- 兼容性更好，但失去了 pnpm 的一些优势

### 4. 工作区包优先

```ini
prefer-workspace-packages=true
```

**目的**：优先使用工作区内的包版本。

**技术背景**：
- Codex 使用 pnpm 工作区（定义在 `pnpm-workspace.yaml`）
- 工作区包含多个包：`codex-cli`, `sdk/typescript`, `shell-tool-mcp` 等
- 当依赖版本同时存在于工作区和 registry 时，优先使用工作区版本

**优势**：
- 开发时自动使用本地修改
- 无需发布到 registry 即可测试跨包更改
- 确保工作区内版本一致性

## 具体技术实现

### 配置优先级

```
1. 命令行参数（--shamefully-hoist）
2. 项目根目录的 .npmrc
3. 用户主目录的 ~/.npmrc
4. 全局配置（/etc/npmrc）
5. 内置默认值
```

### pnpm 工作区配置

```yaml
# pnpm-workspace.yaml
packages:
  - codex-cli
  - codex-rs/responses-api-proxy/npm
  - sdk/typescript
  - shell-tool-mcp
```

`.npmrc` 与工作区的协作：
- `prefer-workspace-packages=true` 确保本地开发使用工作区包
- `shamefully-hoist` 和 `node-linker=hoisted` 影响工作区包的依赖解析

### 与 package.json 的关系

```json
{
  "name": "codex-monorepo",
  "private": true,
  "packageManager": "pnpm@10.29.3+sha512.498e1fb4cca5aa06c1dcf2611e6fafc50972ffe7189998c409e90de74566444298ffe43e6cd2acdc775ba1aa7cc5e092a8b7054c811ba8c5770f84693d33d2dc"
}
```

- `packageManager` 字段锁定 pnpm 版本
- `.npmrc` 配置该版本 pnpm 的行为

## 关键代码路径与文件引用

### 相关文件

1. **pnpm-workspace.yaml**
   - 定义工作区包的位置
   - 与 `.npmrc` 的 `prefer-workspace-packages` 配合使用

2. **package.json** (根目录)
   - 定义 `packageManager` 字段锁定 pnpm 版本
   - 包含仓库维护脚本

3. **pnpm-lock.yaml**
   - 锁定依赖版本
   - 受 `.npmrc` 配置影响生成

4. **codex-cli/package.json**
   - CLI 包的依赖配置
   - 受 `.npmrc` 的 `shamefully-hoist` 影响

5. **sdk/typescript/package.json**
   - TypeScript SDK 配置

### 依赖安装流程

```
1. pnpm 读取 .npmrc 配置
2. 解析 pnpm-workspace.yaml 确定工作区范围
3. 根据 node-linker=hoisted 创建扁平 node_modules
4. 应用 shamefully-hoist 提升依赖
5. 根据 prefer-workspace-packages 链接本地包
6. 生成/更新 pnpm-lock.yaml
```

## 依赖与外部交互

### pnpm 工具

**安装方式**：
```bash
# 使用 corepack（Node.js 16+）
corepack enable
corepack prepare pnpm@10.29.3 --activate

# 或使用 npm
npm install -g pnpm@10.29.3

# 或使用安装脚本
curl -fsSL https://get.pnpm.io/install.sh | sh -
```

**核心命令**：
```bash
# 安装依赖
pnpm install

# 添加依赖
pnpm add <package>

# 添加开发依赖
pnpm add -D <package>

# 运行脚本
pnpm run <script>

# 工作区命令
pnpm --filter <workspace> <command>
```

### 与 npm/yarn 的兼容性

| 特性 | npm | yarn | pnpm (当前配置) |
|------|-----|------|----------------|
| 依赖隔离 | 否 | 部分 | 否（hoisted） |
| 磁盘效率 | 低 | 中 | 高 |
| 安装速度 | 慢 | 中 | 快 |
| 严格 peer deps | 否 | 可选 | 否（已禁用） |

### CI/CD 集成

```yaml
# .github/workflows/ci.yml
- uses: pnpm/action-setup@v4
  with:
    version: 10.29.3
- uses: actions/setup-node@v4
  with:
    node-version: 22
    cache: 'pnpm'
- run: pnpm install --frozen-lockfile
```

## 风险、边界与改进建议

### 潜在风险

1. **依赖提升导致的不确定性**
   - `shamefully-hoist=true` 可能隐藏缺失的依赖声明
   - 某些包可能意外访问到未声明的依赖

2. **Peer dependency 问题被掩盖**
   - `strict-peer-dependencies=false` 允许安装继续
   - 运行时可能出现意外的 peer dependency 错误

3. **工作区包版本冲突**
   - `prefer-workspace-packages=true` 在发布前可能掩盖版本问题
   - 发布后才发现 registry 版本与工作区版本不兼容

### 边界情况

1. **混合包管理器**
   - 如果开发者意外使用 npm/yarn，`.npmrc` 配置被忽略
   - 可能导致 `node_modules` 结构不一致

2. **全局安装**
   - 全局安装的包不受 `.npmrc` 影响
   - 可能导致版本不匹配

3. **嵌套工作区**
   - 复杂的嵌套工作区配置可能导致意外行为

### 改进建议

1. **CI 中启用严格检查**
   ```ini
   # .npmrc.ci（CI 专用配置）
   strict-peer-dependencies=true
   shamefully-hoist=false
   ```
   ```yaml
   # CI 中使用
   - run: pnpm install --config .npmrc.ci
   ```

2. **添加注册表配置**
   ```ini
   # 如果使用私有 registry
   registry=https://registry.npmjs.org/
   @scope:registry=https://private.registry.com/
   ```

3. **添加网络配置**
   ```ini
   # 提高网络超时
   fetch-timeout=60000
   fetch-retries=3
   
   # 使用更快的 registry 镜像
   registry=https://registry.npmmirror.com/
   ```

4. **安全相关配置**
   ```ini
   # 忽略引擎检查（谨慎使用）
   engine-strict=false
   
   # 允许执行 postinstall 脚本（安全风险）
   ignore-scripts=false
   ```

5. **考虑逐步收紧配置**
   ```ini
   # 长期目标：移除 shamefully-hoist
   # 逐步修复依赖声明问题
   # shamefully-hoist=false
   
   # 长期目标：启用严格 peer deps
   # strict-peer-dependencies=true
   ```

6. **添加文档注释**
   ```ini
   # pnpm configuration for Codex monorepo
   # See: https://pnpm.io/npmrc
   
   # Allow hoisting for compatibility with legacy tools
   shamefully-hoist=true
   
   # Disable strict peer deps for smoother installs
   strict-peer-dependencies=false
   
   # Use hoisted node_modules structure
   node-linker=hoisted
   
   # Prefer local workspace packages during development
   prefer-workspace-packages=true
   ```

### 使用示例

```bash
# 安装所有依赖（遵循 .npmrc 配置）
pnpm install

# 查看当前配置
pnpm config list

# 查看特定配置值
pnpm config get shamefully-hoist

# 临时覆盖配置
pnpm install --shamefully-hoist=false

# 工作区操作
pnpm --filter codex-cli add lodash
pnpm --filter sdk/typescript build
```

### 配置对比

```ini
# 当前配置（兼容性优先）
shamefully-hoist=true
strict-peer-dependencies=false
node-linker=hoisted
prefer-workspace-packages=true

# 推荐配置（严格性优先）
shamefully-hoist=false
strict-peer-dependencies=true
node-linker=isolated
prefer-workspace-packages=true
```

选择当前配置的原因：
- 项目可能依赖一些不完全兼容 pnpm 严格模式的工具
- 迁移自 npm/yarn 的过渡期需要
- 某些 IDE 或工具对 pnpm 的隔离模式支持不完善
