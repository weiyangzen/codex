# .bazelignore 研究文档

## 场景与职责

`.bazelignore` 文件是 Bazel 构建系统的忽略配置文件，用于指定哪些目录或文件不应该被 Bazel 扫描和纳入构建图。这在大型 monorepo 中尤为重要，可以避免 Bazel 处理不相关的文件，提高构建性能并减少不必要的依赖分析。

本文件位于 Codex 项目根目录，与 `.gitignore` 功能类似，但专门针对 Bazel 构建系统。

## 功能点目的

### 1. 排除版本控制系统目录

```
.git
```

**目的**：排除 Git 版本控制系统的元数据目录。

**技术背景**：
- Sapling SCM（Meta 开发的版本控制系统）会在 `.git/sl/origbackups` 目录中创建 BUILD.bazel 文件
- 如果不排除 `.git` 目录，Bazel 会递归扫描并尝试解析这些备份文件中的 BUILD.bazel，可能导致构建错误或不一致
- 注释明确指出了这一点："Without this, Bazel will consider BUILD.bazel files in .git/sl/origbackups"

### 2. 排除 Rust 构建输出目录

```
codex-rs/target
```

**目的**：排除 Rust Cargo 的构建输出目录。

**技术背景**：
- `codex-rs/target` 是 Cargo 编译生成的目标文件目录
- 该目录包含大量的 `.rlib`、`.rmeta`、可执行文件等构建产物
- 排除此目录可以显著减少 Bazel 的扫描时间，避免将构建产物误认为源代码

## 具体技术实现

### 文件格式

`.bazelignore` 使用简单的换行分隔格式：
- 每行一个路径模式
- 支持 `#` 开头的注释行
- 路径相对于工作区根目录
- 不支持通配符（与 `.gitignore` 不同）

### 与 .gitignore 的差异

| 特性 | .bazelignore | .gitignore |
|------|-------------|------------|
| 语法复杂度 | 简单（仅路径列表） | 复杂（支持通配符、否定模式等） |
| 递归排除 | 是 | 是 |
| 注释支持 | 是（`#`） | 是（`#`） |
| 通配符支持 | 否 | 是（`*`, `?`, `[]` 等） |
| 否定模式 | 否 | 是（`!`） |

## 关键代码路径与文件引用

### 相关文件

1. **BUILD.bazel** (根目录)
   - 定义了工作区级别的构建规则
   - 与 `.bazelignore` 共同决定哪些文件参与构建

2. **MODULE.bazel** (根目录)
   - 定义 Bazel 模块依赖
   - 与 `.bazelignore` 配合确保外部依赖正确处理

3. **codex-rs/Cargo.toml**
   - Rust 工作区配置
   - `target` 目录由 Cargo 生成，被 `.bazelignore` 排除

### Bazel 扫描流程

```
1. Bazel 启动时读取 .bazelignore
2. 构建文件扫描阶段跳过被忽略的路径
3. 构建图生成时不包含被忽略目录中的目标
```

## 依赖与外部交互

### 与 Sapling SCM 的交互

```
.git/sl/origbackups  # Sapling 备份目录
```

- Sapling 是 Meta 开发的版本控制系统，与 Git 兼容
- 在备份操作中可能在 `.git` 目录内创建 BUILD 文件
- `.bazelignore` 确保这些临时文件不会干扰 Bazel 构建

### 与 Cargo/Rust 工具链的交互

```
codex-rs/target      # Cargo 构建输出
```

- Cargo 和 Bazel 都可以用于构建 codex-rs
- 当使用 Bazel 构建时，需要忽略 Cargo 的输出目录
- 反之，当使用 Cargo 构建时，Bazel 的 `bazel-*` 目录被 `.gitignore` 排除

## 风险、边界与改进建议

### 潜在风险

1. **过度排除**
   - 如果错误地排除了包含源代码的目录，会导致构建失败
   - 当前配置较为保守，仅排除必要的目录

2. **与 .gitignore 不一致**
   - 某些在 `.gitignore` 中的条目可能也需要在 `.bazelignore` 中
   - 例如 `bazel-*` 目录被 `.gitignore` 排除，但 Bazel 本身会自动处理

3. **Sapling 特定问题**
   - 注释提到的 Sapling 问题可能是特定于某些开发环境的
   - 纯 Git 用户可能不理解为什么需要排除 `.git`

### 边界情况

1. **符号链接**
   - Bazel 对符号链接的处理可能因版本而异
   - 如果被忽略的目录是符号链接，行为可能不一致

2. **嵌套工作区**
   - 如果项目包含嵌套的 Bazel 工作区，`.bazelignore` 的行为可能复杂化

### 改进建议

1. **添加更多注释**
   ```
   # Generated by Cargo - not part of source tree
   codex-rs/target
   
   # Sapling SCM may create BUILD files here
   .git
   ```

2. **考虑排除其他常见目录**
   ```
   # Node.js dependencies
   node_modules
   
   # IDE files
   .vscode
   .idea
   ```

3. **定期审查**
   - 随着项目演进，新的生成目录可能需要被排除
   - 建议与 `.gitignore` 定期同步审查

4. **文档化决策**
   - 在 AGENTS.md 或开发者文档中说明为什么需要 `.bazelignore`
   - 特别是 Sapling 相关的背景知识

### 相关配置对比

```
# .bazelignore (当前)
.git
codex-rs/target

# .gitignore (相关部分)
node_modules
bazel-*
.codex/
```

两个文件有明确的分工：
- `.bazelignore`：Bazel 构建优化
- `.gitignore`：版本控制清洁性
