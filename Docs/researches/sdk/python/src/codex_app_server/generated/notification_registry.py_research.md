# sdk/python/src/codex_app_server/generated/notification_registry.py 研究文档

## 场景与职责

该文件是 Python SDK 的核心基础设施组件，负责建立**服务器通知方法名**到**Pydantic 模型类**的映射注册表。它是 app-server JSON-RPC 协议中通知消息反序列化的关键组件，使得 SDK 能够将服务器推送的原始 JSON 通知自动转换为类型化的 Python 对象。

### 核心职责

1. **通知路由**：将服务器发送的 `method` 字段（如 `"turn/completed"`）映射到对应的 Pydantic 模型
2. **类型安全**：提供运行时类型验证，确保通知 payload 符合 schema 定义
3. **扩展点**：作为代码生成目标，当协议新增通知类型时自动更新

## 功能点目的

### 1. 通知模型注册表 (`NOTIFICATION_MODELS`)

```python
NOTIFICATION_MODELS: dict[str, type[BaseModel]] = {
    "account/login/completed": AccountLoginCompletedNotification,
    "account/rateLimits/updated": AccountRateLimitsUpdatedNotification,
    "account/updated": AccountUpdatedNotification,
    "app/list/updated": AppListUpdatedNotification,
    # ... 更多条目
}
```

该字典实现了**方法名到模型类的双向查找**：
- **Key**: JSON-RPC 通知的 `method` 字段（如 `"item/agentMessage/delta"`）
- **Value**: 对应的 Pydantic 模型类，用于验证和解析 `params` 字段

### 2. 模型类导入

文件从 `v2_all.py` 导入了 55 个通知相关的 Pydantic 模型类，涵盖以下领域：

| 类别 | 示例方法 | 模型类 |
|------|----------|--------|
| 账户相关 | `account/login/completed` | `AccountLoginCompletedNotification` |
| 线程生命周期 | `thread/started`, `thread/closed` | `ThreadStartedNotification`, `ThreadClosedNotification` |
| Turn 执行 | `turn/started`, `turn/completed` | `TurnStartedNotification`, `TurnCompletedNotification` |
| Item 进度 | `item/started`, `item/completed` | `ItemStartedNotification`, `ItemCompletedNotification` |
| 消息增量 | `item/agentMessage/delta` | `AgentMessageDeltaNotification` |
| 工具调用 | `item/commandExecution/outputDelta` | `CommandExecutionOutputDeltaNotification` |
| 实时会话 | `thread/realtime/started` | `ThreadRealtimeStartedNotification` |
| 配置警告 | `configWarning` | `ConfigWarningNotification` |
| 错误通知 | `error` | `ErrorNotification` |

## 具体技术实现

### 代码生成流程

该文件完全由 `scripts/update_sdk_artifacts.py` 自动生成，生成逻辑如下：

```python
def generate_notification_registry() -> None:
    out = sdk_root() / "src" / "codex_app_server" / "generated" / "notification_registry.py"
    specs = _notification_specs()  # 从 ServerNotification.json 提取方法名和类名映射
    class_names = sorted({class_name for _, class_name in specs})

    lines = [
        "# Auto-generated by scripts/update_sdk_artifacts.py",
        "# DO NOT EDIT MANUALLY.",
        "",
        "from __future__ import annotations",
        "",
        "from pydantic import BaseModel",
        "",
    ]

    for class_name in class_names:
        lines.append(f"from .v2_all import {class_name}")
    lines.extend([
        "",
        "NOTIFICATION_MODELS: dict[str, type[BaseModel]] = {",
    ])
    for method, class_name in specs:
        lines.append(f'    "{method}": {class_name},')
    lines.extend(["}", ""])

    out.write_text("\n".join(lines))
```

### 源数据提取 (`_notification_specs`)

生成器从 `ServerNotification.json` schema 中提取通知定义：

```python
def _notification_specs() -> list[tuple[str, str]]:
    server_notifications = json.loads(
        (schema_root_dir() / "ServerNotification.json").read_text()
    )
    one_of = server_notifications.get("oneOf", [])
    generated_source = (sdk_root() / "src" / "codex_app_server" / "generated" / "v2_all.py").read_text()

    specs: list[tuple[str, str]] = []
    for variant in one_of:
        props = variant.get("properties", {})
        method_meta = props.get("method", {})
        params_meta = props.get("params", {})

        methods = method_meta.get("enum", [])
        if len(methods) != 1:
            continue
        method = methods[0]

        ref = params_meta.get("$ref")
        if not isinstance(ref, str) or not ref.startswith("#/definitions/"):
            continue
        class_name = ref.split("/")[-1]
        
        # 验证模型类确实存在于生成的 v2_all.py 中
        if f"class {class_name}(" not in generated_source and f"{class_name} =" not in generated_source:
            continue
        specs.append((method, class_name))
```

### 运行时反序列化流程

在 `client.py` 中，`NOTIFICATION_MODELS` 被用于通知反序列化：

```python
def _coerce_notification(self, method: str, params: object) -> Notification:
    params_dict = params if isinstance(params, dict) else {}

    model = NOTIFICATION_MODELS.get(method)
    if model is None:
        # 未知通知类型，包装为 UnknownNotification
        return Notification(method=method, payload=UnknownNotification(params=params_dict))

    try:
        payload = model.model_validate(params_dict)
    except Exception:
        # 验证失败，降级为 UnknownNotification
        return Notification(method=method, payload=UnknownNotification(params=params_dict))
    return Notification(method=method, payload=payload)
```

## 风险、边界与改进建议

### 风险

1. **生成漂移风险**：如果 `ServerNotification.json` 和 `v2_all.py` 不同步（例如手动修改了其中一个），可能导致运行时错误
   - **缓解**：`test_contract_generation.py` 中的 `test_generated_files_are_up_to_date` 测试会在 CI 中检测漂移

2. **缺失通知处理**：如果服务器发送了注册表中未定义的通知，`model` 将为 `None`，通知会被包装为 `UnknownNotification`
   - **影响**：客户端无法访问类型化的字段，只能通过 `params` 字典访问原始数据

3. **验证失败降级**：如果通知 payload 不符合 schema，会静默降级为 `UnknownNotification`
   - **风险**：可能导致难以调试的问题，因为错误被吞掉了

### 边界情况

1. **方法名冲突**：`NOTIFICATION_MODELS` 使用 method 字符串作为 key，如果协议定义了重复的 method 名，后定义的会覆盖先定义的
2. **大小写敏感**：method 名匹配是大小写敏感的（如 `"turn/completed"` ≠ `"Turn/Completed"`）
3. **空 params**：某些通知可能不带 params，此时 `model_validate({})` 的行为取决于模型定义（可选字段需有默认值）

### 改进建议

1. **添加验证失败日志**：在 `_coerce_notification` 中记录验证失败的原因，便于调试：
   ```python
   except Exception as e:
       logger.debug(f"Failed to validate notification {method}: {e}")
       return Notification(method=method, payload=UnknownNotification(params=params_dict))
   ```

2. **方法名常量化**：将 method 字符串提取为常量，避免魔法字符串：
   ```python
   class NotificationMethods:
       TURN_COMPLETED = "turn/completed"
       ITEM_STARTED = "item/started"
       # ...
   
   NOTIFICATION_MODELS: dict[str, type[BaseModel]] = {
       NotificationMethods.TURN_COMPLETED: TurnCompletedNotification,
       # ...
   }
   ```

3. **反向查找支持**：添加从模型类到 method 名的反向查找，便于调试和日志记录：
   ```python
   MODEL_TO_METHOD: dict[type[BaseModel], str] = {
       v: k for k, v in NOTIFICATION_MODELS.items()
   }
   ```

4. **版本兼容性标记**：在生成时添加 schema 版本信息，便于检测协议版本不匹配：
   ```python
   # Auto-generated from schema version: 2.x.x
   NOTIFICATION_MODELS_VERSION = "2.x.x"
   ```

5. **性能优化**：对于高频通知（如 `item/agentMessage/delta`），考虑使用 `lru_cache` 缓存模型验证结果或预编译验证器
